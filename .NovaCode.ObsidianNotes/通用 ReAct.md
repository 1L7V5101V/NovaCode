**通用 ReAct（Reasoning and Acting）** 是大语言模型（LLM）和 AI Agent 领域最经典、最底层的核心演进范式之一。它由谷歌和普林斯顿大学的研究者在 2022 年底提出，核心思想是**将大模型的“推理（Reasoning）”与“行动（Acting）”无缝结合**。

通俗来说，ReAct 赋予了 AI 一种“边想边做，做完再想”的思维能力，让它不再只是盲目地单向输出文本，而是能够像人类一样通过“思考 $\rightarrow$ 采取行动 $\rightarrow$ 观察结果 $\rightarrow$ 再次思考”的循环来解决复杂问题。

## 1. ReAct 的核心三步循环（Thought-Action-Observation）

一个通用的 ReAct 框架在接收到复杂任务后，会进入一个标志性的死循环，直到问题被解决：

```
[用户目标] ──> 思考 (Thought) ──> 行动 (Action) ──> 观察 (Observation) ──> 思考... ──> [最终答案]
```

- **🧠 思考 (Thought)：** 模型对当前局势进行分析、推理，规划下一步应该做什么，或者解释为什么上一步失败了。
    
- **🛠️ 行动 (Action)：** 模型决定调用外部工具（Tools）。例如：执行一段 Python 代码、查询数据库、或者调用谷歌搜索。
    
- **👁️ 观察 (Observation)：** 工具执行后返回的**客观结果**（如代码报错、搜索出来的网页片段）。这个结果会被重新喂回给大模型，作为它的“短期记忆”。
    

## 2. 经典案例对比：传统 LLM vs ReAct

假设用户问一个需要实时信息和计算的复杂问题：

> **“当前（2026年）英伟达的最新显卡型号是什么？它的官方售价折合人民币是多少？”**

### ❌ 传统 LLM (直接生成)

大模型会直接检索自己的训练权重，由于知识断层，它可能会胡编一个答案（幻觉），或者直接回答：“我的知识截止到 2023 年，我不知道 2026 年的最新型号。”

### ReAct 范式

ReAct 会在内部展开如下的精彩对白：

> - **Thought:** 用户想知道 2026 年英伟达最新显卡型号和人民币售价。我需要先通过搜索获取最新的显卡发布信息。
>     
> - **Action:** `search["NVIDIA latest graphics card 2026 release price"]`
>     
> - **Observation:** _（工具返回搜索结果）_ “英伟达于 2026 年初发布了 RTX 5090 显卡，官方建议零售价（MSRP）为 1999 美元。”
>     
> - **Thought:** 我已经知道了型号是 RTX 5090，价格是 1999 美元。现在我需要查询当前的美元对人民币汇率，以便进行折算。
>     
> - **Action:** `search["USD to CNY exchange rate 2026"]`
>     
> - **Observation:** _（工具返回搜索结果）_ “当前美元兑人民币汇率为 1 比 7.21。”
>     
> - **Thought:** 价格是 1999，汇率是 7.21。我需要计算 $1999 \times 7.21$。我应该调用计算器工具确保计算精准。
>     
> - **Action:** `calculator["1999 * 7.21"]`
>     
> - **Observation:** _（工具返回结果）_ `14412.79`
>     
> - **Thought:** 所有信息已齐备。型号是 RTX 5090，折合人民币约为 14413 元。可以向用户汇报了。
>     
> - **Final Answer:** 2026年英伟达最新的显卡型号是 RTX 5090，其官方售价为 1999 美元，按照当前 7.21 的汇率折合人民币约为 14,413 元。
>     

## 3. 为什么通用 ReAct 具有划时代的意义？

在 ReAct 提出之前，学术界有两条平行的技术路线：

1. **思维链（CoT, Chain of Thought）：** 引导模型写下“一、二、三”的推理步骤，但这只是大模型内部的闭门造车，无法获取外部新知识。
    
2. **行动驱动（Act-only）：** 类似于早期的自动脚本，直接机械化地调用 API，缺乏应对突发错误和动态调整的能力。
    

**ReAct 的伟大之处在于“协同”：**

推理（Thought）为行动（Action）指明方向；行动带来的观察（Observation）又反过来纠正推理的偏差。当行动遇到报错（如代码编译失败）时，模型能在下一轮的 Thought 中自我反思：“哦，刚才函数名写错了，我应该换个写法。”

## 4. 通用 ReAct 的局限性与演进

虽然 ReAct 是极其优秀的基石范式，但在工业级 AI Agent（如你之前提到的 Coding Agent）开发中，纯粹的、通用的 ReAct 面临很大的挑战：

- **Token 消耗黑洞：** 因为每一步循环都要把历史所有的 Thought、Action、Observation 叠加上去，对话轮次一多，上下文窗口瞬间爆炸。
    
- **易陷入死循环：** 如果工具连续返回错误（Observation），模型容易卡在同一个错误的 Thought 里出不来（即“降智”或陷入僵死状态）。
    
- **缺乏大局观：** ReAct 是一种典型的**贪心算法思维**（走一步看一步），它很难在项目开始前做好长期的宏观架构规划。
    

**未来的演进：**

正因如此，现在的框架（如 LangGraph）往往不再直接使用纯粹的通用 ReAct。开发者们更倾向于使用**结构化的状态机（State Graph）**——用图节点（Nodes）固化宏观的业务流程，只在局部某个需要动态调测的节点内部，引入 ReAct 思想来解决具体问题。
