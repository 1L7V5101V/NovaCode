### 3. checkpoint / resume 机制是怎么做的
> 我们的 checkpoint/resume 机制，核心要解决的是三个场景：**prompt 超预算被迫裁剪后、任务中断需要恢复、工作区发生变化**——这三种情况下，agent 能接着干，而不是从头重来。
> 
> 先说 checkpoint 的数据结构。它不是一个单独的文件，是嵌在 session JSON 里的一个字段。每次创建 checkpoint，我们会把当时的**用户目标、下一步建议、所有接触过的文件路径和它们的 SHA256 指纹、以及完整的运行时配置**——模型是什么、approval policy 是什么、max_steps 是多少、甚至 feature flags 是什么——全部打一个快照存起来。相当于是给 agent 的运行现场拍了一张照片。
> 
> 那什么时候打这个快照呢？六个时机：
> 1. **每次调完工具之后**——这是最核心的。意味着即使中间崩溃，最多只丢一步。
> 2. 模型返回最终答案时
> 3. 到达步数上限被迫停时
> 4. 触发 budget 裁剪时
> 5. 恢复时发现文件变了
> 6. 恢复时发现配置变了
> 
> 第 5 和第 6 个其实是恢复过程中的保护性 checkpoint。
> 
> 然后说 resume。用户通过 `--resume session_id` 从命令行恢复一个 session。CLI 会调 `Pico.from_session()`，把之前落盘的 session JSON 完整加载回来。初始化过程中有一道关键检测叫 `evaluate_resume_state()`，它干三件事：
> 
> **第一步**，检查所有文件摘要的 freshness——我们在 memory 里存的每个文件摘要都附带一个创建时刻的 SHA256。如果磁盘上的文件哈希变了，就把这个过期摘要直接删掉，不让模型看到旧信息。
> 
> **第二步**，检查 checkpoint 里记录的 key_files 的哈希跟当前磁盘文件是不是一致。如果变了，标记为 `partial-stale`。
> 
> **第三步**，把 checkpoint 里保存的运行时配置跟当前进程的配置逐字段对比。11 个字段，包括 model、approval_policy、feature_flags、workspace_fingerprint 这些。如果配置变了，标记为 `workspace-mismatch`。
> 
> 最终判定出四种状态：`full-valid` 表示啥都没变；`partial-stale` 表示文件变了；`workspace-mismatch` 表示运行时配置变了；`schema-mismatch` 表示代码版本升级，不兼容了。
> 
> 这些状态不是内部静默处理的。我们会把 checkpoint 信息渲染成一段文本注入到下一轮 prompt 的 system prefix 里。模型能看到："当前恢复状态是 partial-stale，过期的文件有 src/main.py，已完成步骤有……，下一步建议是……"。这样模型自己就知道哪些文件需要重新读、哪些信息已经不可靠了。
> 
> 但这里面有一个关键的设计抉择：checkpoint 信息只是**提示**，不是**指令**。模型可以选择不采纳。真正的防御在于——`partial-stale` 状态下过期的文件摘要已经被我们从 memory 里删掉了，模型在 prompt 的 memory section 根本看不到旧内容，只能通过 `read_file` 重新获取。这样一来，即使模型想偷懒复用旧信息，它也没处可拿。
> 
> 最后说我们的 benchmark 验证。我们在 12 个固定回归任务里有 3 个专门测恢复的：一个测 context reduction 触发的 checkpoint 能不能正确创建，一个测文件被外部修改后恢复能不能正确识别 stale 并重新读，一个测换了运行时配置后能不能正确标记 workspace mismatch。通过 scripted model output 做确定性测试，保证这些恢复逻辑是可复现的，不是靠模型运气。
> 
> ---
> 
> **如果有追问："那你们怎么防误信旧状态？"**
> 
> 两个层面。第一层，文件 freshness 不匹配时，memory 里对应的文件摘要直接清除，模型看不到旧内容。第二层，checkpoint 在 prompt 里只是文本提示，模型不采纳也不会死——它后面反正还会读文件，读到新内容后 memory 自然就更新了。


### 4. 目前项目进度是什么样的，部署在哪里
项目目前处于**内部实验验证阶段**，没有对外开源或部署到公网服务。

- **部署形态**：跑在我们本地的 GPU 服务器上，主要对接本地部署的**Qwen3.5** 27B. 主要用来处理组内的代码仓库数据分析任务。-实验数据清洗 + 论文图表复现：组里攒了一批电化学测试数据，格式不统一，以前靠 MATLAB 脚本一把处理。现在 agent 读原始 CSV、写清洗脚本、输出统一格式的表格和对比图，过程中如果清洗规则不生效还能自己重读数据调整逻辑。
