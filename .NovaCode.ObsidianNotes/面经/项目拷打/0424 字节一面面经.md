# 如何判断当前任务是否结束以及结束的标志

![](https://cdn.nlark.com/yuque/0/2026/png/65082237/1777038115315-db30d99e-7b11-4084-b834-120def81d07f.png)这个问题其实要考量两个地方：**一个是这轮agent运行什么时候停下来，另一个是用户交给它的工程任务到底算不算完成。前者回答的是这轮 agent 执行有没有进入终态，后者回答的是用户要的结果有没有被验证过。**

运行层面，Pico 有一个主循环。用户发起任务以后，系统会创建一份任务状态，里面记录这次运行当前处于什么状态、已经调用了几次工具、最近一次工具是什么、最后为什么停下来，以及最终回复是什么。**任务状态会被系统单独记录下来。**

每一轮循环里，Pico 会把当前仓库信息、历史记录、记忆和检查点整理给模型，然后看模型下一步想做什么。如果模型要读文件、改文件、跑命令，那这轮还没结束。系统会先检查工具能不能用、参数是否合法、有没有重复调用、需不需要审批，然后执行工具，把结果写回上下文，再进入下一轮。

所以工具调用在 Pico 里只是推进任务，不是结束任务。**真正的正常结束，是模型给出一条可以被系统识别的最终回复。这个时候 Pico 会把任务状态写成“已完成”，同时把停止原因写成“正常返回最终回答”**。也就是说，Pico 判断一次运行正常结束，看的是结构化状态，并不是来自模型的输出“我完成了”

这样做的好处是，它把“当前状态”和“停止原因”分开了。当前状态告诉你这次运行是还在跑、已经完成、被迫停止，还是失败；停止原因告诉你它为什么停，是正常给出最终回答，还是步数用完，还是模型连续输出不合法。这样后面排查问题的时候，不会把所有停止都混成一类。

**如果模型一直没有给出最终回复，Pico 也不会无限跑**。比如工具调用次数到了上限，它会停下来，并记录成“步数达到上限”；如果模型连续输出空内容或者格式不对，它会记录成“重试次数达到上限”。这些都算运行停止，但不算任务成功完成。

还有一个细节是，工具失败本身不会直接结束任务。比如文件路径不对、补丁参数错了、命令执行失败，Pico 会把这个失败结果写回下一轮，让模型有机会根据错误继续修正。只有正常给出最终回复，或者触发明确的停止边界，这次运行才会结束。

但到这里还只是“运行结束”。工程任务有没有完成，还要看验收证据。Pico 的固定评测里，每个任务都会定义干净的测试仓库、允许使用的工具、步数预算、期望产物和校验脚本。最后判断通过时，不只看智能体有没有回复完成，还会看几个条件：有没有超出步数预算，校验脚本有没有通过，目标文件或产物是否存在，以及停止原因是不是正常返回最终回答。

**这样设计是为了防止一种很常见的问题：模型说自己完成了，但文件没改对，测试没过，或者只是因为步数耗尽被迫停下。Pico 会把这些情况区分开。正常停下只是必要条件，真正通过还要有外部验证。**

所以总结一下：Pico 对任务结束的判断分成两层。运行时看主循环有没有进入明确终态，正常终态是任务状态变成已完成，并且停止原因是正常返回最终回答；评测层再看结果有没有被校验脚本、目标产物和步数预算验证过。这样“完成”就不是模型的一句主观总结，而是一组可以追踪、可以复盘、可以评测的系统状态。

# CC内部也有一个非常严格且分层的上下文的治理，然后你的pico这种上下文的治理跟他的是如何保证不冲突的?

我理解 Claude Code 的上下文治理，本质上是在解决一个问题：code agent 工作久了以后，prompt 里会混进太多不同性质的信息，比如项目规则、Git 状态、工具输出、对话历史、长期记忆和当前任务进度。如果都塞进上下文窗口，不仅贵，而且容易把真正重要的任务状态冲掉。

它的设计主要就是分层处理。固定规则，比如 CLAUDE.md、用户规则、项目规则、Git 初始状态，会在会话开始时作为稳定上下文注入。当前任务里的消息、工具调用、工具结果进入主循环，但每轮发模型前会先做清理：大的工具输出先落盘，只保留路径和预览；旧工具结果通过 microcompact 清掉；窗口接近上限时再触发 autocompact。长期记忆用 MEMORY.md 做索引，具体内容拆成 topic 文件，需要时再按相关性预取。

我觉得最关键的是部分是 Session Memory。它不是普通的“快满了再总结”，而是后台定期维护一份当前任务笔记，记录任务做到哪里、涉及哪些文件、踩过什么坑。等 compact 真的发生时，系统优先用这份 Session Memory 作为压缩后的摘要，再保留最近还没被总结的消息。这里用 lastSummarizedMessageId 做游标，知道哪些历史已经进 notes，哪些还必须保留，同时还会避免拆散 tool_use 和 tool_result。所以它的压缩是平时持续维护状态，最后在窗口压力大的时候复用这个状态。

![](https://cdn.nlark.com/yuque/0/2026/png/65082237/1777041814518-0a524ebe-1c2d-4c7b-a768-43514be33055.png)

**pico如何预防上下文治理时的冲突：**

pico 预防上下文冲突的核心做法，是不给所有信息都塞进一个 summary，而是给每类信息明确归属。

工具结果先进入 history 和 trace，作为可回看的原始过程；memory 只沉淀少量高价值事实，比如最近文件和文件短摘要。这个摘要不是事实源，它带 freshness，文件变了就失效，写文件和 patch 也会主动清掉旧摘要。

任务推进状态不放进 memory，而是放在 TaskState 和 checkpoint 里，记录 attempts、tool_steps、last_tool、stop_reason、next_step、key_files 这些信息。durable memory 只收长期稳定事实，并且会过滤 secret、noisy output 和 transient task state，避免把短期进度写成长期记忆。

最后由 ContextManager 统一装箱，把 prefix、memory、relevant memory、history、current request 分段放进 prompt。超预算时按固定顺序裁：先 relevant memory，再 history，再 memory，最后 prefix；current request 不裁。

所以pico预防冲突主要是在runtime 先把信息分层、失效、过滤、排序。这样旧摘要不会压过当前文件，长期记忆不会混进短期任务状态，历史日志也不会把当前请求挤掉。

# 相关记忆的召回你是怎么保证召回的是稳定的内容或者任务?

![](https://cdn.nlark.com/yuque/0/2026/png/65082237/1777041194287-73742322-c388-41e4-a7d7-8e7cebef7bdb.png)

这块地方的难点是召回以后怎么保证它不会污染当前任务。尤其代码场景里，过期记忆比没有记忆更危险。没有记忆最多是多读几次文件；过期记忆可能让模型基于错误假设去改代码。

在pico里面是把记忆做了分层，没有设计成一个单纯的语义搜索库

**第一类是项目级稳定记忆**，比如这个项目常用什么测试命令、代码风格是什么、重要目录怎么组织。这类记忆生命周期比较长，但也要能被新的证据覆盖。

**第二类是任务级记忆**，比如当前任务目标、已经尝试过的方案、哪些测试失败、哪些文件改过。这类记忆和当前会话强绑定，任务结束后不能随便带到另一个任务里。

**第三类是文件级记忆**，比如某个文件的大致职责、关键函数、之前读过的摘要。这类记忆必须和文件的新鲜度绑定。文件一旦被修改，旧摘要就不能继续当作可信事实。

在召回时，我会优先做几件事。

**第一，先限定作用域。**当前仓库、当前分支、当前任务相关的记忆优先，跨项目记忆默认不召回。因为很多污染都来自作用域过宽。

**第二，优先精确匹配，**再做语义召回。比如当前任务已经涉及某几个文件，那文件路径、模块名、测试名这种强信号，比泛泛的语义相似度更可靠。语义召回可以作为补充，但不能作为唯一依据。

**第三，召回结果要带来源。**记忆最好不是一句孤立结论，而要能知道它来自哪个文件、哪次运行、哪条测试结果、哪个任务阶段。没有来源的记忆，我会降低权重，最多当作提示。

**第四，关键修改前要回读原始证据。**比如记忆说‘这个模块负责权限校验’，模型不能直接基于这句话去改，应该再打开对应文件确认。也就是说，记忆提供方向，不提供最终事实裁决。

**第五，文件相关记忆要有失效机制。**只要文件发生写入、补丁修改，或者工作区状态变化明显，旧的文件摘要就应该失效。否则你会遇到一个很隐蔽的问题：模型以为自己知道文件内容，其实文件已经变了。

这里的核心取舍是召回率和精确率。召回做得激进，模型会显得很聪明，能记住很多历史；但风险是把不相关或过期的信息带进当前任务。召回做得保守，可能会多读文件、多跑搜索，成本高一点，但正确性更稳。对代码智能体，我会偏向精确率，因为代码修改是有副作用的，错一次可能会破坏项目。

所以对pico对设计时遵循一个原则是：**记忆不是事实本身，而是帮助模型定位事实的索引**。真正关键的判断，还是要回到当前工作区、当前文件内容和当前测试结果。这样记忆既能减少重复探索，又不会把系统带进过期上下文。

# 如果把产品需求文档扔进来之后，它能自动实现、自动检索评估、最后完成测试。如果让你设计这样一套执行框架，你会怎么做？

**【这问题其实就是面向软件工程任务的 Agent Harness 怎么设计】**

**你可以告诉面试官，这个场景其实是面向软件工程任务的 Agent Harness 怎么设计，然后向他分享你最近也看过很多harness实践，有一些自己的理解，模型在这之中就是起个推理作用。**

**针对这个问题，目前的业务场景上，需求文档通常是不完整的**，里面有业务目标、用户场景、隐含约束，也有很多没有写清楚的验收标准。直接让模型读完就改代码，风险非常高。所以在这里我想设计一套Harness框架，并把它拆成多个阶段来解决这个问题。

**第一个阶段是需求结构化**。系统拿到需求文档后，先不改代码，而是把它拆成目标、非目标、用户流程、数据约束、界面变化、接口变化、验收标准、风险点。这里最关键的是把模糊需求变成可检查的清单。比如‘支持用户导出报表’这个需求，要进一步拆成导出入口在哪里、导出格式是什么、权限怎么控制、空数据怎么处理、失败怎么提示、是否需要测试覆盖。

**第二个阶段是仓库理解**。执行框架要先读项目说明、目录结构、现有测试、构建命令、相似功能实现，而不是凭空写代码。这里可以用搜索先定位候选模块，再读取关键文件。这个思路和现在代码智能体常见的按需上下文策略是一致的：先保留路径、命令、查询结果这些轻量索引，需要时再加载具体内容，不用把整个仓库一次性塞进上下文。

**第三个阶段是计划生成和人工确认。**对于小改动，可以让系统自动继续；对于涉及数据库、权限、支付、删除数据、大规模重构的需求，必须进入人工确认。计划里要写清楚预计修改哪些文件、增加哪些测试、有哪些不确定点。如果需求文档本身有矛盾，系统应该优先提出澄清问题。

**第四个阶段是受约束执行。**真正执行时，模型不能随便读、随便改、随便跑命令。它只能通过受控工具来做事，比如搜索、读取、修改文件、运行测试命令。所有工具都要有边界：读文件要限制路径，写文件要记录差异，执行命令要限制超时和危险操作，涉及高风险变更要审批。这一层其实是 Harness 很核心的地方。模型负责推理和决策，但执行权不能完全裸奔。每一步工具调用都应该留下轨迹，包括它为什么读这个文件、为什么改这段代码、命令执行结果是什么。文件修改前最好有检查点，至少要能通过版本控制回滚。这样即使任务失败，也能知道失败在哪里，而不是得到一堆不可复盘的改动。

**第五个阶段是自动评估。**除了跑测试，还要做多层评估。第一层是静态检查，比如格式、类型、语法。第二层是单元测试和相关测试。第三层是需求验收清单逐项对照。第四层是变更审查，比如有没有改到不该改的文件，有没有引入明显安全风险，有没有破坏兼容性。必要时还可以让另一个评审角色只看需求和差异，不参与实现，专门挑遗漏。

**第六个阶段是收尾报告**。它要告诉用户：实现了哪些需求，修改了哪些地方，运行了哪些测试，哪些测试通过，哪些没有运行，原因是什么，还有哪些风险需要人工看。**这才是这个流程唯一人工review的地方，人不需要从头盯着模型每一步执行，但要能在最后快速判断**：这次变更是否符合需求，验证是否充分，风险是否可接受。如果发现问题，也能根据运行轨迹回到具体步骤，而不是重新猜模型做了什么。

如果让我把这套能力落到 pico 上，我会先补三块能力：

第一，补一个**需求文档解析和验收清单生成模块**。让系统能把自然语言需求拆成目标、约束、任务项和可检查标准，而不是直接进入代码修改。

第二，补一个更明确的**任务状态机和检查点机制**。比如需求解析、仓库理解、计划生成、执行中、验证中、待人工确认、已完成、部分完成、失败这些状态要明确。每次重要变更前后都要能保存状态，方便回滚和复盘。

第三，补一个更系统的**评测器**。它不能只记录工具调用次数，还要判断需求覆盖率、验证充分性、风险等级和是否存在未解决项。
