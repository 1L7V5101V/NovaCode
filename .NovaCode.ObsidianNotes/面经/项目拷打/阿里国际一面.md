## 2. 长上下文治理怎么做的？如果用户只输入一个“hi”，prompt长什么样，会占用多少token？

> [!NOTE]
> 长上下文治理核心是 **"分 section 设预算 + 按优先级裁剪"**（`context_manager.py`）。
> 
> prompt 分 5 个 section，每个有独立的 **budget（上限）** 和 **floor（底线）**：
> 
> | section | budget | floor | 裁剪优先级 |
> |---------|--------|-------|-----------|
> | prefix | 3600 | 1200 | 最后裁 |
> | memory | 1600 | 400 | 第三 |
> | relevant_memory | 1200 | 300 | **最先裁** |
> | history | 5200 | 1500 | 第二 |
> | current_request | 不限 | 不限 | **永不裁** |
> 
> total budget = 12000 字符。超了就从 `relevant_memory` 开始砍，砍到 floor 还不够就砍 history，依次类推，直到总长度 <= 12000。`current_request` 永远不动。
> 
> 每条 `relevant_memory` 笔记还会按 `per_note_budget` 做 `_tail_clip`（`context_manager.py:269`），超限的尾部截断加 `...`。
> 
> ---
> 
> ### 用户只输入 "hi" 的 prompt 长什么样
> 
> 首次运行、无记忆、无历史时（假设 workspace 有基本信息）：
> 
> ```
> You are pico, a small local coding agent working inside a local repository.
> 
> Rules: [14 条规则, ~400字]
> ...
> Tools: [7 个工具描述 + 示例, ~1000字]
> ...
> Workspace:
> - cwd: D:\.Projects\xxx
> - branch: main
> - status: clean
> - recent_commits: ...
> 
> Memory:
> - task: -
> - recent_files: -
> - file_summaries: -
> - episodic_notes: 0
> - durable_topics: -
> 
> Relevant memory:
> - none
> 
> Transcript:
> - empty
> 
> Current user request:
> hi
> ```
> 
> **大约 2450~2950 字符，约 610~740 tokens**。远低于 12000 的 total budget，不会触发任何裁剪。
> 
> 占用大头是 **prefix（规则 + 工具描述 + workspace 信息）**，约 2300~2800 字符，占整体的 90%+。记忆、历史、当前请求加起来不超过 150 字符。


## 3. 模型接入做了哪些？如果要用本地部署的模型，模型应该部署在本地还是其他服务器上？优缺点是什么？

> 我们接了四类后端，统一成一个 `complete(prompt, max_new_tokens)` 接口：
> 
> 1. **Ollama**——`OllamaModelClient`，跑本地的 qwen、deepseek-coder 这些。
> 2. **OpenAI-compatible**——`OpenAICompatibleModelClient`，走 `/v1/responses` 接口，支持 prompt cache，适配了 SSE 流式响应和普通 JSON 响应两种返回格式。
> 3. **Anthropic-compatible**——`AnthropicCompatibleModelClient`，走 `/v1/messages`，DeepSeek 也是通过这条路接的。
> 4. **FakeModelClient**——测试用，不进网络，直接按预定的输出序列返回。
> 
> **大语言模型部署在本地服务器。**
> 
> **优点**：
> - GPU 利用率高——服务器可以同时服务多个人。
> - 对本地开发机没有要求
> - 模型版本和配置集中管理，换模型改服务器端就行。
> 
> **缺点**：
> - 网络延迟，不适合对实时性要求很高的交互场景。
> - 如果局域网断了或者服务器在跑训练显存不够，模型就调不动。
> 
> **如果反过来把大语言模型部署在在本机**：
> - 优点是零延迟、完全离线。
> - 缺点是显存资源紧张——本地工作站通常就一张卡，模型和别的任务抢资源；而且对实验室这种几个人共用的环境，每个人都拉一份模型显得比较浪费。

## 4. 当前的设计是否支持在一个session里面切换模型？如何设计？
>**当前不支持**，有两层原因。
>
>第一层是 `runtime_identity` 的约束。每个 checkpoint 里都记录了运行时身份——10 多个字段，包括 `model` 和 `model_client`。如果在一个 session 中间切了模型，`evaluate_resume_state()` 会发现 `model_client` 字段不一致，直接标记为 `workspace-mismatch`。这意味着切换模型会触发一轮"配置漂移"的恢复流程，checkpoint 里保存的下步建议会被模型谨慎对待。这本身不是一个 bug，但在当前设计里，模型切换是一个"值得警惕的事件"，不是一个常规操作。
>
>第二层是 prompt cache。`OpenAICompatibleModelClient` 会把 prefix 的 hash 作为 prompt_cache_key 传到后端。如果中间换了模型，这个 cache key 就失效了，但这不是大问题。更大的问题是——不同模型的工具调用格式偏好不同。我们的 parse 层同时支持 JSON 和 XML 两种格式，但不同模型在这两种格式上的稳定性不同。同一个 session 里切模型，历史里可能混合着 JSON 格式的调用记录和 XML 格式的调用记录，模型的 few-shot 上下文会被搞乱。
>
>**但如果真要设计，我会怎么做？**
>**接口层面**
>加一个 `Pico.switch_model(new_client)` 方法，调用时会做三件事：
>1. 把当前状态打一个 final checkpoint，记下切换前的 runtime_identity（包括旧模型名）。
>2. 清空 `prompt_cache` 相关的 [[metadata]]，因为新模型的缓存不共享。
>3. 在 history 里插入一条系统事件 `"model switched from gpt-4o to deepseek-coder"`，让模型感知到上下文的变化。
>
>**checkpoint 层面**
>我可能会加一个新的 resume 状态叫 `model-switched`，介于 `partial-stale` 和 `workspace-mismatch` 之间。它的含义是："checkpoint 里目标的 next_step 仍然可以继续，但因为模型变了，历史中的工具调用格式不再可靠参考。"这个状态下，checkpoint text 在 prompt 里会带上一条额外的说明——"模型已切换，后续工具调用请按当前模型的格式输出。"
>
>**记忆层面**
>切换模型后，memory 里的文件摘要仍然有效——因为文件内容没变，变化的只是模型。所以 memory 不需要清空。但 `episodic_notes` 中那些 tagged 为"之前模型犯过的错、需要绕开的坑"的 process notes，可能在新模型上不适用。我会在切换时由调用方决定是否保留这些 notes。
>
>**更重要的约束**
>我会在实现时做一条硬约束：**不允许在工具执行中途切换模型**。也就是 `switch_model` 只能在两次 `ask()` 之间调用，不能在 agent 跑了一半的时候换模型。这个约束本身是合理的——你也不会在一个人写到一半的时候换他的 IDE，对吧？
>
>**总结**：当前不支持是因为 `runtime_identity` 和 checkpoint 一致性的设计假设了单一模型。如果真的要做，我会把它设计成 session 内一个显式的事件，而不是一个属性的替换——让模型知道模型换了、让 checkpoint 记录这次切换、让 memory 选择性保留。

## 5. 如果需要在一个session里面切换不同的模型继续完成任务，会有什么影响？
> 这个问题的本质是：模型变了，之前 session 里积累的那些状态还有多少是能用的？我从四个维度来讲影响——哪些能用、哪些不能用、哪些需要小心处理。
> 
> **第一，memory 和文件摘要——基本不受影响。** 
> memory 里存的是"task_summary: 给 main.py 加一个二分查找函数"和"sample.txt 的内容摘要：前三行是 alpha、beta、gamma"。这是关于文件内容的事实，不是关于模型的事实。所以切换模型后，memory 完全不需要清空。新模型一启动就能看到当前任务目标和已经读过的文件摘要，不需要从头扫一遍仓库。
> 
> **第二，prompt cache——失效，但不是大问题。**
> 我们的 agent 每轮调用模型时，prompt 里有一段稳定不变的部分——系统指令、工具列表、工作区描述，这部分叫 prefix。变的部分只有：memory 状态、历史记录和当前请求。我们把系统 prompt 的 [[hash]] 当 prompt_cache_key 传给后端。换了模型后，这个 key 对应的是旧模型的缓存，新模型不共享。所以切完模型后第一轮调用不会有 cache 命中，消耗更多token，延迟会高一截。但后面的轮次会重新建立新模型的缓存。
> 
> **第三，checkpoint 状态——变成 workspace-mismatch，但影响可控。**
> 当前代码里，`evaluate_resume_state()` 发现 `model` 字段变了，会把恢复状态标为 `workspace-mismatch`。checkpoint 里记录的 "next_step: 在 sample.txt 中把 beta 改成 beta-locked" 其实在新模型上仍然是可以执行的——文件没变，文件哈希也没变，只是模型变了。但当前状态机**优先判了 workspace-mismatch，没有给模型这个 signal**。
> 
> 影响是：checkpoint text 会注入 prompt，但状态是一个偏警告级的 "workspace mismatch"，模型看了可能会过于谨慎——比如重新读一遍已经读过的文件。实际上它应该被当成一个更温和的状态：**"模型换了，checkpoint 里的目标仍然有效，但之前模型留下的工具调用格式不要学。"**
> 
> 所以在影响层面，这里有一个**过度报告的问题**——报告了一个不应被视为错误的漂移。
> 
> **第四，也是最值得警惕的影响——历史记录中的工具调用格式。**
>  这是最大的坑。session history 里存了旧模型生成的工具调用记录，比如旧模型习惯用 JSON 格式：
> ```
> [tool:patch_file] {"path": "sample.txt", "old_text": "alpha", "new_text": "alpha-locked"}
> ```
> 
> 新模型在生成回复时，会看到这段历史作为 [[few-shot 上下文]]。如果新模型本身偏好 XML 格式，但看到历史里全是 JSON，它就可能会混用两种格式，或者对自己不确定的格式产生混淆。这会影响 parser 的稳定性——虽然我们的 parse 同时支持两种格式，但如果模型输出一个格式混搭（比如 `<tool>{"name":"patch_file","args":...}` 里面又包了 XML 内容），parse 可能会失败，触发 retry 循环。
> 
> 这是影响最大的部分，而且在当前设计里没有好的自动解法。只能靠 **checkpoint 里明确告诉模型"模型已切换，历史中的工具格式仅供参考"**，让模型自己警觉。
> 
> **第五，process notes——部分失效。**
> 
> memory 里存了一类叫 process note 的东西，比如 "write_file partial_success on README.md; inspect diff before retry"。这个笔记本身是模型无关的——不管谁来跑，这个工具结果都一样。但另一种 notes 比如 "run_shell 执行 pytest 时失败原因是 pytest 版本不兼容"——这个教训对不同模型的价值是一样的。所以 process notes 不需要清空，只是新模型可能用不同的方式来避免同一个问题。
> 
> **总结：**
> 


> | 维度 | 影响程度 | 原因 |
> |---|---|---|
> | memory / 文件摘要 | 无影响 | 存的是文件事实，不是模型行为 |
> | prompt cache | 短暂影响 | 第一轮无缓存，延迟略高 |
> | checkpoint 状态 | 过度警告 | 目标仍有效，但被标为 workspace-mismatch |
> | 历史工具格式 | 高影响 | 新旧模型格式偏好不同，导致 few-shot 混乱 |
> | process notes | 低影响 | 工具结果是模型无关的，可继续引用 |
> 
> 核心结论是：**切换模型的真实影响不在状态恢复上，而在历史格式的混合上。** 如果要支持这个能力，最需要解决的其实不是 checkpoint 状态机，而是怎么在 history 里标注"模型边界"，让新模型知道哪些历史是自己的、哪些是别人的。
