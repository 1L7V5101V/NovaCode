# 1. 每个项目问的都是在这个项目学到最多的是什么。

> [!NOTE]
> 从这个项目学到最有价值的几点：
> 
> 1. **分层记忆设计**：working → episodic → durable 三层，每层有不同的生命周期和压缩策略。episodic 靠关键词召回、durable 用文件持久化 + subject_key 去重覆盖。这和 LangChain 那种"一股脑塞进向量库"的思路不同——对 coding agent 场景，结构化的确定性记忆比 embedding 召回更可靠、更可审计。
> 
> 2. **Checkpoint 的新鲜度契约**：每个 checkpoint 记录了 key_files 的 `sha256` freshness，resume 时检测文件是否被外部修改、runtime identity 是否变化。这个"不信任旧摘要"的设计比简单存 session history 更诚实 —— LLM 最大的问题就是自信地消费过时信息。
> 
> 3. **上下文预算的节流策略**：`ContextManager` 定义固定优先级（relevant_memory → history → memory → prefix），超预算时按顺序压缩，且每个 section 有 floor 防止被完全挤掉。这比"截断末尾 N tokens"的暴力方案工程上合理得多。
> 
> 4. **防御性编码习惯**：`RunStore._write_json_atomic()` 用临时文件 + atomic replace 防半截 JSON；`validate_tool()` 在 `run_tool()` 之前独立校验参数；`repeated_tool_call()` 挡掉最简单的 LLM 死循环。这些都是和 LLM 打过交道才会加的 guard。
> 
> 5. **代码本身就是文档**：每个核心函数顶部写明了 "为什么存在 / 输入输出 / 在链路中的位置" —— 这在开源项目里极其少见，比任何外部文档都更有价值。新人读 `runtime.py` 的 `ask()` 和 `run_tool()` 就可以理解整个 agent 控制流。
> 
> 核心 lessons：agent 框架的核心竞争力不在于 prompt 技巧，而在于**状态管理、安全护栏、和资源节制**。

> ---
> 
> ## 一、pico 的定位与现状
> 
> pico 是一个**纯 Python 标准库实现**的轻量终端 coding agent，零外部依赖。支持 Ollama / OpenAI / Anthropic / DeepSeek 四种后端。核心架构：
> 
> ```
> CLI → Pico.ask() → ContextManager(拼prompt) → ModelClient → parse() → run_tool() → checkpoint/memory
> ```
> 
> 已有亮点：checkpoint 恢复、层状记忆(working→episodic→durable)、上下文预算裁剪、secret 脱敏、sub-agent delegate。
>

> 
> ## 二、主要改进空间
> 
> ### 2.1 模型交互层
> 
> | 问题 | 影响 | 建议 |
> |------|------|------|
> | 自定义 XML/JSON 解析 tool call (`pico\runtime.py:1212-1261`) | 脆弱、不支持原生 function calling、token 浪费 | 接入 OpenAI/Anthropic 原生 tools API |

# 3. pico问了上下文管理里面，历史记录具体是如何放入的，有没有改进空间，有没有了解claude code、cursor等工具是如何管理上下文的？
> [!NOTE]
> ### 历史记录放入上下文的机制
> 
> 核心流程分三步：
> 
> #### 1. 记录（`runtime.py:444`）
> 
> ```
> Pico.record(item) → session["history"].append(item)
> ```
> 
> 每次用户消息、工具结果、assistant 回复都**无脑追加**到 `session["history"]` 列表，从不主动删除旧条目。
> 
> #### 2. 渲染时压缩（`context_manager.py:361-405`）
> 
> `_compressed_history_entries()` 每轮遍历全部历史，按新旧做不同处理：
> 
> |分区|条件|处理方式|
> |---|---|---|
> |**最近 6 条**|`index >= len(history)-6`|保留全文，每行上限 900 字符|
> |**旧 `read_file`**|重复读同一文件|折叠为文件摘要行（复用 `LayeredMemory`）|
> |**旧 `run_shell`**|其他工具|仅保留命令+输出前 3 行|
> |**旧对话**|user/assistant 消息|裁剪至 60 字符|
> 
> #### 3. 从后往前塞入预算（`context_manager.py:319-344`）
> 
> 从最新条目开始反向填充，超预算时：
> 
> - **最近条目**：继续裁剪至可用空间（不低于 20 字符）
> - **旧条目**：裁剪至 20 字符
> 
> 如果总 prompt 仍然超过 `total_budget=12000`，按 `relevant_memory → history → memory → prefix` 顺序缩减各 section 预算，再重新渲染。
> 
> ---
> 
> ### 改进空间
> 
> 1. **历史列表无限增长** — `session["history"]` 只增不减，长会话后每轮遍历数千条。应在压缩渲染后，将旧历史**物理截断**为摘要条目。
>     
> 2. **基于字符 vs 基于 token** — 中文字符数/ token 比例与英文不同，`12000` 字符对中文 prompt 更易超限。建议改用 tokenizer 计算真实 token 成本。
>     
> 3. **固定最近窗口 = 6** — 没有语义优先级。可能某个旧消息比最近几条更关键。可引入 embedding 相关性排序或 LLM 重排。
>     
> 4. **无多轮摘要** — 旧条目只是截断而非摘要。可以定期让 LLM 把一段旧对话压缩成一条语义摘要，替换原始 N 条记录。
>     
> 5. **`run_shell` 摘要太粗暴** — 只取前 3 行拼接，丢失结构化信息。对大输出（如 `ls -R`、`find`）可按行数/类型智能摘要。
>     
> 6. **`read_file` 去重仅对同一路径** — 如果用户交替读文件 A、B、A，中间的那个 A 会被保留，旧的 A 被折叠。但重叠是简单的路径匹配，没有考虑**内容相似度**。
>     
> 7. **无 token 预算动态调整** — 各 section 的预算比例是固定的（history 默认 5200）。可以根据实际观测到的每轮历史消耗比例，动态调整预算分配。

> [!NOTE]
> 像 Claude Code、Cursor、Windsurf 等现代 AI 编程工具，普遍采用了一套“收集 $\rightarrow$ 检索与过滤 $\rightarrow$ 压缩与重构 $\rightarrow$ 动态组装”的管道（Pipeline）机制。
> 
> ## 1. 全局索引与静态分析（基础建档）
> 
> 当你首次在 Cursor 或 Claude Code 中打开一个项目时，它们会在后台静默启动一个**索引引擎**。
> 
> - **向量化存储（Vector Embeddings）：** 工具会将代码库中的文件拆分成较小的代码块（Code Chunks），通过 Embedding 模型将其转化为向量并存储在本地（如 SQLite/Chroma 等）。这样，当你想找“处理用户登录的逻辑”时，即使不记得确切文件名，系统也能通过语义计算匹配到相关代码。
>     
> - **抽象语法树（AST）解析：** 仅靠向量检索是不够的。这些工具还会对代码进行 AST 静态分析，提取出项目中的**类、函数、变量及其依赖关系**（类似于 IDE 的 Go to Definition 功能）。
>     
> - **尊重配置（`.gitignore` 等）：** 它们会自动忽略 `node_modules/`、`.git/`、编译产物以及大体积的二进制文件，确保上下文的纯净度。
>     
> 
> ## 2. 智能上下文检索（根据意图抓取）
> 
> 当你输入一句“帮我修复这个 Bug”或者“实现一个新接口”时，工具会启动多路召回（Multi-route Retrieval）机制：
> 
> - **当前活动状态（Active State）：** 必然包含你**当前打开的文件**、**光标所在的位置（选中区域）**、以及最近浏览过的几个标签页。
>     
> - **语义检索（RAG）：** 将你的 Prompt 转化为向量，去本地向量数据库中捞取最相关的代码片段。
>     
> - **相关性图谱（Graph-based）：** 如果你引用了 `User` 类，系统会顺着 AST 依赖图谱，自动把 `User` 类的定义以及它依赖的 `BaseModel` 也顺带捞出来，哪怕你并没有打开这些文件。
>     
> - **报错信息与终端输出：** 像 Claude Code 这种运行在终端的 Agent，会自动捕获上一条命令的失败输出（如 `npm run test` 的报错），并将其作为最高优先级的上下文。
>     
> 
> ## 3. 用户显式指引（Explicit Context）
> 
> 除了自动检索，优秀的工具都提供了“划重点”的功能，让用户直接干预上下文：
> 
> - **Cursor 的 `@` 机制：** 允许用户精确指定 `@Files`（文件）、`@Folders`（文件夹）、`@Codebase`（全局搜索）、`@Web`（实时联网搜索 API 文档）或 `@Git`（最近的 Commits/Diffs）。
>     
> - **Claude Code 的文件权限：** 作为命令行 Agent，它会通过主动询问或配置文件，明确自己被允许读取和修改哪些文件目录。
>     
> 
> ## 4. 上下文压缩与智能重构（Sparsification）
> 
> 捞出来的东西太多，塞不下怎么办？工具会对内容进行“脱水”：
> 
> - **骨架模式（Skeleton/Signatures）：** 如果一个关联文件有 1000 行，但你只需要其中一个函数的逻辑，系统不会把 1000 行全传过去。它会把文件压缩成一个“大纲”，只保留类名、函数签名（名、参数、返回值）以及相关的那个函数体，中间无关的代码用 `// ... existing code ...` 代替。
>     
> - **Jina Reader / Markdown 转换：** 如果涉及网页或长文档，工具会将其转化为最省 Token 的干净 Markdown 格式。
>     
> 
> ## 5. 动态 Prompt 组装与 System Prompt 设计
> 
> 最后，所有被筛选、压缩后的有效信息，会被按照特定的结构化格式（通常是 Markdown 或 XML 标签）拼接到 System Prompt 和 User Prompt 中。
> 
> 例如，Claude 特别喜欢 **XML 标签**（因为 Anthropic 模型对 XML 极其敏感）：
> 
> XML
> 
> ```
> <system_instructions>
> 你是一个资深的软件工程师...
> </system_instructions>
> 
> <current_file_context path="src/auth.ts">
> // 当前正在编辑的代码
> </current_file_context>
> 
> <referenced_snippets>
> <snippet path="src/models/user.ts">
> export interface User { id: string; name: string; }
> </snippet>
> </referenced_snippets>
> 
> <user_query>
> 帮我把当前文件里的登录函数改为异步，并返回 User 类型。
> </user_query>
> ```
> 
> ## 总结：两者的微小差异
> 
> 虽然底层逻辑相似，但在交互产品形态上略有不同：
> 
> - **Cursor (Copilot/Chat 模式)：** 更倾向于“被动响应 + 交互引导”。它在前端 UI 上做了极佳的优化（如 `@` 符号、Inline Edit），让用户非常直观地看到并控制 AI 正在看什么。
>     
> - **Claude Code (Agent 模式)：** 更倾向于“主动探索”。由于它运行在终端且拥有工具执行权（Tool Calling），它在遭遇未知时，会自己通过 `grep`、`find`、`view_file` 等内置工具在你的项目里反复横跳、自主寻找上下文，直到它认为拿够了信息，再一并提交给大模型求解。

# 4. pico的上下文管理和ClaudeCode等工具相比有什么优缺点？为什么会有上下文长度限制？上下文过长问题是什么？
> [!NOTE]
> ## Pico vs Claude Code
> 
> | 维度 | Pico | Claude Code |
> |------|------|-------------|
> | **压缩层级** | 单层：渲染时一次性截断/摘要 | **五层流水线**：budget reduction → snip → microcompact → context collapse → auto-compact |
> | **历史存储** | 无限增长，只增不删 | 物理截断 + 多级压缩策略 |
> | **预算单位** | **字符数**（12000 chars，对中文不友好） | **Token 数**（200K-1M tokens） |
> | **缓存意识** | 无 | 精心设计——compact 复用主对话 cache key，避免 98% cache miss |
> | **LLM 摘要** | 无 | **结构化 9-section 摘要**（intent, 技术决策, 文件, 错误, 待办等） |
> | **压缩后恢复** | 无 | 重新读取最近 5 个文件 + 恢复 todo/plan 状态 + continuation 指令 |
> | **子代理** | 简单 `history_text()` 传递 | 独立上下文窗口 + delta 增量摘要 |
> | **工具结果处理** | 旧 `run_shell` 取前 3 行拼接 | **hot tail + cold storage**，大结果存磁盘仅留引用 |
> | **read_file 去重** | 同一路径折叠，复用 memory 摘要 | 多策略：tool result trimming → 占位符替换 |
> | **成本优化** | 无 | **`cache_edits`** 手术式删除旧工具结果，不破坏 prompt cache |
> 
> ## Pico vs Cursor
> 
> | 维度 | Pico | Cursor |
> |------|------|--------|
> | **上下文策略** | 渲染时压缩历史 | **RAG + @-mentions**：embeddings 检索代码 + 手动指定文件 |
> | **代码库感知** | 无独立索引，依赖 memory 记录路径 | **全量代码库索引**（embeddings），跨文件语义检索 |
> | **长会话处理** | 历史只压缩不删除 | 无内置 compact，建议手动开新会话 |
> | **上下文透明度** | `_metadata()` 记录各 section 长度 | **Curosr 3.3+ 上下文环**：可视化 tokens 分布（rules/skills/MCP/files） |
> | **规则持久化** | 无（memory 中的 episodic notes 有限） | `.cursorrules` / `.cursor/rules`：全局 + glob-scoped 规则 |
> 
> ---
> 
> ## Pico 的主要优势
> 
> 1. **实现简单直接** — 单文件 509 行，逻辑清晰，容易理解和修改
> 2. **分 section 预算管理** — prefix/memory/relevant_memory/history/current_request 各自的 budget 和 floor，可独立调整
> 3. **去重折叠** — 同一 `read_file` 路径自动折叠为 memory summary，减少重复信息
> 4. **开源可控** — 所有策略都在自己手里，可以自定义任何行为
> 
> ## Pico 的主要不足
> 
> 1. **无 token 感知** — 使用字符数而非 token 数做预算，中英文混合场景不准确
> 2. **无 LLM 摘要** — 旧历史只有截断没有语义摘要，信息丢失严重
> 3. **无 [[prompt cache 感知优化]]** — 每次压缩都 invalidate cache，成本更高
> 4. **无压缩后恢复** — 压缩后不重新注入关键文件/状态，模型"失忆"
> 5. **历史列表无限增长** — 物理上从不删除，遍历成本随会话线性增加
> 6. **固定最近窗口 6** — 没有按语义/相关性动态调整
> 7. **子代理有独立窗口，但[[父子之间的上下文传递通道太窄]]（300 字符无结构截断），且缺乏增量摘要和结构化 handoff 机制。**
> 8. **无结构化摘要模板** — Claude Code 的 9-section 模板确保关键信息不遗漏
> 
> ---
> 
> ## 最大改进建议
> 
> 从差距来看，**最值得优先实现的三个方向**：
> 
> 1. **引入 tokenizer-aware 预算** — 用 `tiktoken` 等替代 `len()`，避免中英文字符偏差
> 2. **增加 LLM 摘要层** — 历史超预算时，先让模型生成结构化摘要（含 intent/决定/待办），替换旧 N 条记录；摘要后**重新读取最近文件**恢复工作状态
> 3. **物理裁剪 + 分层存储** — 旧历史渲染后可从 `session["history"]` 移除，保留到 `session["archived"]`；渲染时只遍历活跃部分
