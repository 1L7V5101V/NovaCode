# 2、在做本地代码智能体时，你采取了哪些措施来解决多轮任务中的“上下文膨胀”问题？

![697](https://cdn.nlark.com/yuque/0/2026/png/65082237/1776953359299-762c56be-701a-450c-b680-0bf62b346b1e.png)

我觉得多轮任务里的‘上下文膨胀问题’并不是一个单纯的 token 问题，我更倾向把它看成**一个状态管理失控的问题**。因为真正会膨胀的，不只是对话长度，而是系统把很多不同层级的信息全都塞进了同一个提示里：比如用户目标、历史讨论、已经验证过的事实、失败过的尝试、工具输出、文件内容、还有已经失效的中间推理。这些东西混在一起，模型的注意力会被稀释，成本和延迟会上升，更严重的是它会开始抓错重点。

在pico里，我才用了上下文治理的思路，把上下文拆成四层。

**第一层是热上下文**，也就是当前这一步真正要用的工作集。比如当前子任务、最近一次工具调用结果、正在修改的 2 到 5 个文件、还没解决的约束。这一层必须严格控预算，因为它直接进提示。

**第二层是结构化任务状态**。这个比自然语言历史更重要。比如总目标、约束条件、当前计划做到哪一步、哪些假设已经被证伪、哪些文件被改过、当前失败的测试用例是什么。这里我尽量用结构化字段。这样可以避免丢失当前任务状态的问题发生。

**第三层是可更新的工作记忆**，也就是摘要记忆。我在实现的时候没有把所有历史压成一段摘要，而是沉淀成可复用的稳定事实，比如用户偏好、关键设计决策、已经确认的根因、不能碰的边界条件。它是增量更新的，不是每轮都从头总结。这样做的好处是不会出现摘要漂移，也就是摘要越写越像，但逐渐偏离真实执行轨迹。

**第四层是冷数据**，也就是完整执行轨迹和运行工件。这个层我会完整保留，用来审计、回放和失败恢复，但默认不进提示。只有在需要追责、复盘或者恢复某个分支状态时，才按需检索回来。  
整个上下文治理的核心设计是：每一轮不是追加历史，而是重建上下文。也就是根据当前任务状态，从记忆层、工作区、运行痕迹里，重新装配一份最小但足够的上下文。

**（对比其他方案 trade-off）**

如果只做滑动窗口截断，系统最简单、最便宜，但很容易忘掉早期约束，特别是长任务中前面说过一次、后面很久才生效的条件。  
如果只做摘要，成本会降很多，但会有信息压缩损失，而且摘要一旦写偏，后面所有轮次都会在错误记忆上继续推。  
如果只做检索增强，它适合召回静态知识，不适合管理动态任务过程；因为多轮任务里最关键的往往不是知识，而是当前状态变化。  
如果你把状态结构化得特别重，鲁棒性会提升，但工程复杂度会上去，结构设计、状态迁移、脏状态修复都会变贵。  
**（从trade-off引出我们的思路）**

所以pico把上下文分层后做了一个折中的方案：  
高价值、高稳定性的信息做结构化；  
中价值、可压缩的信息做摘要化；  
低价值但需要追溯的信息做归档化。  
然后由上下文管理器在每一轮按任务意图去装配提示。

我还通过benchmark来检测我这套治理有没有效，但我也不会只看 token 降了多少。我会看四个指标：  
第一，任务成功率有没有掉；  
第二，约束遗漏率有没有下降；  
第三，长任务中断后恢复成功率怎么样；  
第四，单位任务的成本和延迟是否可控。  
如果一个系统只是把提示变短了，但恢复后经常忘记用户限制、经常改错文件，那不叫解决上下文膨胀，只是把问题藏起来了。  
**（回顾总结一下，给面试官记忆点，方便追问）**

我总结一下，对于多轮任务的上下文膨胀，不能简单的靠压缩对话内容可以解决，pico是把‘对话系统’升级成‘状态机 + 记忆系统 + 检查点系统’。”

# 3、如果遇到特别大的工程文件（比如一个超过一万行的Java或C++类文件），导致读取时上下文直接超限，这种情况有什么办法解决？你觉得应该提供给大模型哪些特定的工具来处理大文件或代码搜索场景？

### 怎么解决读大文件

遇到特别大的类文件，肯定不能想着把整个文件都塞进大模型，我觉得应该思考**怎么让模型高效地找到重点、理解重点、再验证重点**。因为这种文件，就算是工程师自己看，也不会从头到尾硬读，而是先根据报错、调用链、最近改动这些线索定位到关键区域，再去精读相关方法和上下文。所以我觉得，解决思路不是全文喂给模型，而是让模型先定位，再分段阅读，再做摘要，最后做验证。

比如一个一万行的类，**真正和当前问题相关的，可能就只有其中两三个方法，外加它上下游调用的几段代码**。那我肯定不会让模型把整份文件全读一遍，而是先帮它把这几段找出来。

**我的整体思路是从整体到局部，一步步缩小范围**，具体来说，我会把处理流程拆成五层。

**第一层是仓库级缩小范围**。很多时候并不是这个一万行文件的每一行都重要，而是只有几个热点区域重要，比如报错栈打到的方法、最近一次改动涉及的函数、某个接口实现、某个状态机分支、某个宏展开后的逻辑。所以第一步不是把一到一万行整份读进来，而是先看报错堆栈、测试失败信息、代码变更、关键字搜索命中、调用关系、引用关系，把范围缩到几个符号或者几个代码块。

**第二层是文件内的结构化切块**。按程序结构切：对 Java 就按类、字段、方法、注解、继承关系来切；对 C++ 就按命名空间、类、方法、模板、宏代码块、头文件依赖关系来切。即使这个文件没有很好地抽象语法树解析，我也会退化到用符号索引、语法树工具或者规则匹配先拿到目录，再决定读哪里。

**第三层是渐进式纰漏**。这个是借鉴claude code阅读skill的做法。模型第一次看到的不是全文，而是一个文件提纲：有哪些类、哪些方法、每个方法大概多少行、谁引用它、它最近有没有变动、复杂度高不高。然后模型再发起第二跳：读这个符号的签名、注释、局部实现；第三跳：读它的调用方和被调用方；第四跳：必要时再补相邻代码块。这样上下文里保留的是结构上下文 + 局部正文，不用去读整份源码。

**第四层是摘要和记**。大文件场景里，摘要的主要目的是降低下一轮成本。比如我刚读完一个三百行方法，我会缓存一个机器可读的摘要：这个方法做什么、关键状态变量、前置条件、异常路径、依赖了哪些成员字段。下一轮如果还要用到，就优先带摘要，不再带全文。但这个摘要必须绑定文件哈希或符号哈希，一旦代码改了就失效，不然很容易把旧认知带进下一轮，越跑越偏。

第五层是验证闭环。大模型即使定位到了问题，也不能直接凭感觉改一万行的大而全类。正确做法是：只在局部符号或局部代码块上生成补丁，然后跑最小验证集，比如编译、相关单元测试、受影响路径的静态检查或者关键字检查。如果失败，再把失败信息反哺给模型。这一点很重要，因为大文件里“看懂”和“改对”是两回事。

### 应该提供给大模型哪些特定的工具来处理大文件或代码搜索场景

分几类吧。

第一类是**导航工具**，比如文件提纲、符号列表、按符号读取、按代码块读取、读取相关符号。（比如ctags、tree-sitter）

第二类是精确检索工具，比如正则搜索、查找符号、查找引用、查找调用方、查找被调用方、查看类型继承层次（比如grep、ripgrep、git grep）

第三类是语义辅助工具，比如语义搜索，但我不会让它单独决策，而是作为文本检索之后的重排。

第四类是编辑工具，比如按符号打补丁、重写方法体、预览差异，最好还能接支持语法树的精确修改。（核心目的就是：别整文件重写，只改该改的地方）

第五类是验证工具，比如运行定向测试、编译单文件或指定模块、编辑后识别受影响测试。

第六类是记忆工具，比如生成符号摘要、带时效性的摘要缓存、过期摘要失效。这套工具给了以后，模型面对大文件就不再是“瞎读”，而是“先导航，再钻取，再验证”。

（这里面其实有几个很重要的权衡。第一个权衡是固定分块和结构化分块。固定分块实现最简单、语言无关，但语义边界会断；结构化分块更符合代码理解，但需要抽象语法树、语言服务或者至少符号索引工具支持，工程复杂度更高。第二个权衡是文本检索和语义检索。语义检索对“我只知道大概意思，不知道精确名字”的场景很有帮助，但代码搜索里噪声很大，尤其 C++ 这种模板、宏、重载多的语言，很多时候精确符号命中比语义相似更可靠。所以我会做混合检索，但排序上让精确匹配、符号命中、调用图邻近性优先。第三个权衡是重型语言服务和通用降级方案。Java 用完整语言服务通常体验很好，但 C++ 经常受编译选项、宏、头文件依赖影响，语言服务不是永远可用。所以我的设计一定是分层降级：优先语言服务，不行就语法树或符号索引，再不行就全文搜索和规则匹配，不能把能力全压在最重的一层。第四个权衡是预建索引还是按需索引。全仓预建索引响应很快，适合大仓库和高频使用，但会有初始化成本、索引陈旧和运维问题；按需索引更轻，但第一次查询可能慢。所以我会按仓库规模来选，中小仓库按需，大仓库做增量索引。第五个权衡是单智能体和多智能体。多智能体可以并行探索几个热点符号，但会有协调成本和结论漂移，所以我一般只让子智能体做只读调查，不让它们各自独立写代码。）

如果让我继续做 `pico`，我会保留它现在这条受约束工具 + 上下文压缩 + 轻量记忆 + 检查点的路线，但优先补三件事。第一，给它加按符号感知的导航层，不要只靠按行读文件。第二，给它加引用关系、调用图、类型层次这一类代码搜索工具，不要只靠文本搜索。第三，给它加带时效性的摘要缓存和大文件策略路由：文件一旦超过某个阈值，就自动走先看提纲的流程，而不是让模型碰运气去反复读片段。这样它面对超大 Java/C++ 文件时，才会从勉强能读升级成稳定可用。

### claude code是怎么做的

我之前看 Claude Code 再处理这个问题的时候，有一个很细致的流程：

claude code会先用 Glob 找候选文件，再用 Grep 缩到相关方法、类名、报错点这些热点位置，再优先用 LSP 做定义跳转、引用查找和调用链导航，只把真正相关的几个代码片段交给 Read 按 offset + limit 局部读取；如果读取范围太大，Read 会直接报超限，逼模型继续缩小范围，而不是自动截断一大段内容塞进上下文。与此同时，已经读过且没变化的内容不会重复灌入窗口，旧的 Read、Grep、shell 输出在上下文接近上限时还会被 compact 压缩或清掉，所以它本质上是在用搜索、跳转、局部精读、上下文回收这套链路解决大文件问题，这样模型面对一万行类文件时，不是从第一行读到最后一行，而是像工程师一样，先定位，再精读。

# 4、针对“工具重复读取、工具无效调用”的问题，你大概做了哪些处理措施？（追问：如果大模型使用read工具附带参数读了四次同一个文件，这算重复读取吗？）

**其实这个题不是缓存优化问题，是一个Agent Loop的问题。“工具重复读取”和“工具无效调用”，本质上都不是某一次提示词没写好，而是模型在多轮任务里容易出现的两种退化行为。**

第一种，是没有获得新信息，却还在反复读取同样的内容。

第二种，是动作本身没有意义，比如参数不合法、路径越界、空调用、明知道会失败还继续调。

这两类问题如果不治理，表面上看是浪费几次调用，实际上会带来三个后果：一是推理轨迹变长，二是上下文被无效信息挤占，三是模型会在局部循环里越走越偏。

所以我在pico里，对这个问题的处理思路是把它当成一个系统性治理问题，分成四层来做：先减少没必要的读取，再在执行前做硬校验，再在执行后做记忆和压缩，最后把这些行为纳入观测和评测闭环。做到“前面预防、中间约束、后面消化、最后可量化”。

**第一层的做法是减少没必要的读取。**

我的一个基本判断是，很多重复读取，其实不是因为模型故意浪费，而是因为一开始没有拿到足够好的导航信息。它不知道应该读哪里，所以就会不断试探。因此，真正有效的做法，不是上来就让它读大量文件，而是先给一个很小但重要的上下文文信息。

比如仓库当前状态、最近改动、核心说明文档、配置文件、项目入口这类信息，先让模型知道这个代码库大概是什么结构。与此同时，让搜索类工具优先承担“探路”的角色，先低成本缩小范围，再决定要不要真正读取文件内容。这样做的核心思想是：搜索应该比全文读取便宜，导航应该早于钻取。

这里有一个很明确的权衡。你让模型先搜索、再决定读文件，会多一步决策；但换来的好处是，真正大段读取文件的次数会下降，整体成本和噪声都更低。我会认为这是值得的，因为多轮任务里最怕的不是少一次读取，而是没有方向地乱读。

**第二层，是执行层的硬约束。**

治理无效调用，不能只靠提示词教育模型，而要靠runtime的约束。

也就是说，任何工具调用，在真正执行之前，都要经过统一检查：这个工具是不是允许调用，参数是不是合法，调用是不是重复，是否需要审批，通过之后才能执行。

这块的核心思路把错误尽量前置。比如文件读取时，要检查文件是否存在、读取区间是否合理；搜索时，不能让它传一个空模式；执行命令时，要限制超时；写文件时，不能把目录当文件写；做文本补丁时，必须精确命中目标，不能模糊替换；委派任务时，任务内容不能为空。

**第三层，是对重复读取的分层定义。**

很多人容易把“重复读取”讲得过于粗暴，好像只要是同一个文件再读一遍就是问题。但我想分两点来看这个问题

第一点是执行层重复。它看的是：是不是同一个工具、同样的参数、连续调用，而且中间没有任何新的信息增量。只有这种情况，我才会把它当成典型坏循环，并在运行时直接拦住。因为这种调用几乎可以确定没有价值，本质上只是模型卡住了。

第二点是语义层重复。它看的是：虽然不一定是完全相同的调用，但历史上是不是已经反复读取过同一路径、同一主题、同一段信息，导致上下文里出现大量重复内容。这一层我通常不在执行时强拦，而是在上下文整理阶段去处理，把旧的重复读取折叠掉，把真正有价值的新读取保留下来。

我之所以要分这两点，是因为如果你把“重复读取”定义得太绝对了，比如只要同一个文件第二次读取就算重复，那**会误杀很多合法行为**。比如长文件本来就需要分段读；模型读完上半段，再读下半段，这不是坏事。再比如文件被修改之后，再重新读一遍确认修改结果，这也不是坏事。甚至有时候模型第一次读得太粗，第二次带着明确问题回看某一段，也可能是合理的。

不是“同一个文件又读了一次”就叫重复，而是“同样的动作、没有新增信息、仍然连续发生”，这才是我要重点治理的重复。

**第四层，是上下文压缩和摘要复用。**

即使执行层已经拦住了一部分显然无意义的重复，历史轨迹里仍然会留下很多“相似但不完全相同”的读取记录。这个时候，如果你什么都不做，模型下一轮看到的上下文还是会越来越臃肿。所以我会在上下文管理层做两件事。

第一件事，是把较旧的、按路径或主题高度相似的读取记录折叠起来，不让它们一遍遍占据上下文窗口。第二件事，是尽量把已经读过的文件内容沉淀成简短摘要，在后续轮次里优先复用摘要，而不是反复把原文塞回去。

这里的关键不是缓存全文，而是缓存提炼后的结论。因为全文缓存虽然省读取成本，但在代码场景下很危险，一旦文件改过，旧内容就可能变成过期信息。相比之下，短摘要加失效机制更稳。也就是：文件一旦被写入、被补丁修改，和它相关的摘要就应该立刻作废，不能再被当成可信记忆继续复用。

**第五层，是观测和评测。**

如果只做规则，那没办法落地，所以需要评测和观测把这些行为变成可记录、可统计、可回放的指标。

也就是说，每次运行都要能看到：这轮一共调了多少次工具，多少次是参数非法，多少次是重复调用，多少次历史读取被折叠，多少次复用了文件摘要。这样你才能知道，这套治理到底有没有真正减少噪声，而不是只是感觉上更优雅了。

### 追问

这个问题不能直接回答是或者不是，应该分层定义。

如果这四次调用在执行层面是完全一样的，比如同一个文件、同一段范围、参数完全一致，而且中间没有任何新信息，也没有文件发生变化，那我会把它定义为重复读取，而且应该在运行时尽早拦住。

但如果虽然是同一个文件，读取范围不一样，比如第一次读前两百行，第二次读后两百行；或者中间文件已经被改过，所以模型重新读取确认结果；或者模型这次是带着一个更具体的问题回看局部内容，那我不会简单把它定义成坏重复，因为它仍然可能带来信息增量。

不过即便如此，在上下文层我还是会把旧的同路径读取尽量折叠掉，并优先复用摘要，避免同一文件内容反复占用上下文。也就是说，我对这个问题的答案不是二元的：执行层判断的是‘完全重复动作’，上下文层治理的是‘重复信息堆积’。

### 总结（帮助面试官回忆你的内容，更好提问）

对我处理‘工具重复读取、工具无效调用’时，会把它拆成三个问题：

第一，怎么让模型更少走到那一步；

第二，走到了以后，怎么低成本、低误杀地拦住；

第三，拦住以后，怎么把错误反馈、上下文压缩、记忆失效和评测指标全部打通。

# 5、你觉得现在大家都在讲的围绕大模型的“Harness”工程，它到底是一个什么概念/东西？

Harness的中文翻译是马具，Harness工程直译的话我觉得可以翻译成驾驭工程。

它的本质上就是把只会预测下一个词元的大模型，变成能在真实环境里持续做事的系统的那整套外骨骼。

它至少要解决几件事：给模型清晰的目标和边界，给它可调用的工具，管理状态和记忆，驱动它在一个执行循环里持续运行，记录全过程，并且在必要的时候拦住危险动作。OpenAI 对Agent的简单定义，其实也是指令、安全边界和工具；同时它又把构建智能体的基本原语总结成模型、工具、状态或记忆、编排，并强调智能体的核心是一个持续运行的执行循环。把这些放在一起看，其实就是 harness。

我觉得它被讨论的多，很火，是因为大家慢慢发现，真正难的不是让模型会回答，而是让它在多轮、多工具、长流程里持续保持方向感。如果用一个日常ai coding的实例来解释的话，就是想在没有人类介入的情况下，让agent持续高效的完成目标，比如让codex/claude code持续跑很长的时间。

我之前看Anthropic的博客里写的是，他们发现长时任务里智能体常见的失败并不神秘：要么一上来就想一步到位，结果把上下文撑爆；要么做到一半没有留下清晰交接，下一轮只能靠猜；要么其实没做完，却提前宣布成功。他们在这里对应的 harness 解法也非常工程化：第一轮先用一个初始化智能体把环境搭起来，产出功能清单、进度文件、初始化脚本和初始提交；后续的编程智能体每次只做一个功能点，先读取进度和历史提交，先做基本的端到端验证，再开始改动；结束时必须把环境留在一个干净状态，并留下结构化的更新记录。

目前最新的博客文章里，它们又进一步强调结构化产物、上下文重置，以及“规划—生成—评审”这种把生成和审核拆开的结构。

我从中学到的是 harness 是对会话边界、交接物、验证节奏和失败恢复的系统设计。

再往深一层看，harness 还决定模型怎么用工具。Anthropic 专门写过一篇怎么给智能体设计工具，核心结论很直接：工具更多，并不等于效果更好，工具要有清晰目的、好名字、好描述、好返回格式。它们还发现，在复杂的串行决策里，一个很简单的思考工具都能让模型更稳，因为模型终于有了一个在拿到工具结果之后停下来核对规则、补齐信息、检查路径的地方。所以我认为工具不是给得越多越好，而是要被 harness 收敛成一个可解释、可审计、可回滚的动作边界。

很多人一谈 harness，就自动想到多智能体，但我觉得这是一个很常见的误区。OpenAI 的建议其实非常明确：先把单智能体压到极致，再决定要不要拆成多智能体；触发条件通常是提示逻辑太复杂、工具重叠严重，或者任务天然可以并行。Anthropic 对研究场景的结论也类似：多智能体在广度优先式探索、并行搜索、多工具、多上下文窗口的任务上收益很大，但代价是词元和成本显著上升，并不适合所有编程任务。所以 harness 的成熟，不是你有几个智能体，而是你知不知道什么时候该保持单体，什么时候该拆成编排系统。

未来两年，模型层当然还会持续进步，但业务真正的护城河，越来越可能长在 harness 上。因为模型能力会越来越商品化，而你怎么组织上下文、怎么设计工具、怎么做审批和回滚、怎么做追踪和评测、怎么把失败变成下一轮系统改进，这些才是企业里真正难抄走的工程资产。

所以我才想去做pico这个项目，来真正实践一下harness工程。目前它已经把一个最小 harness 的骨架拆出来了：运行时模块对应执行循环，工具模块对应工具层，上下文管理和记忆模块对应上下文与长期信息，任务状态和运行存储模块对应状态持久化，评估和指标模块对应评测与度量，工作区模块对应工作边界。
