# 4. 这个项目里最核心、最难的工程问题是什么？

> [!NOTE]
> 从我阅读整个代码库来看，最核心也最难的工程问题是 **"长上下文治理"**——或者说：**在固定 token 预算下，如何决定 prompt 里该放什么、不该放什么，以及砍掉什么不会坏事儿。**
> 
> 具体难在哪几个方面：
> 
> ##### 1. 多目标冲突
> prompt 由 5 个 section 组成（`context_manager.py:28`）：
> ```
> prefix → memory → relevant_memory → history → current_request
> ```
> 每个 section 都在抢预算：
> - **prefix** 丢了 → agent 不知道自己是干什么的
> - **memory** 丢了 → 忘了长期约定
> - **relevant_memory** 丢了 → 重复读文件
> - **history** 丢了 → 失去对话上下文连贯性
> - **current_request** **不能丢**（用户当前消息如果被裁了，整个对话就断了）
> 
> 预算就那么多（默认 12000 字符），每个 section 有 floor（底线），裁剪有顺序（先砍 relevant_memory，再砍 history ...）。如何编排这套优先级本身就是个系统设计问题。
> 
> ##### 2. 裁剪粒度难以控制
> 不是简单"超了就砍尾巴"，而是：
> - 每一条 `relevant memory` 笔记需要按 `per_note_budget` 做 `_tail_clip`（`context_manager.py:269`）
> - 历史记录按条裁剪，不能从中截断
> - 要保证裁剪后的输出仍然可读、不出现半句话
> 
> ##### 3. 非线性依赖
> 某个 section 的压缩放大到另一个 section 的行为变化。比如 relevant_memory 被裁太狠 → agent 不记得事实 → 多调一次 `read_file` → 历史更长 → prompt 更超预算。这是一个**正反馈循环**，需要做合理的降级策略来打破。
> 
> ##### 4. 为什么这比模型接入、工具调用更难
> 模型接入是标准化的（OpenAI / Anthropic / Ollama 协议都有现成 client），工具调用的 schema 声明也不复杂。但上下文预算管理**没有现成的库可以抄**，它是这个 harness 自己设计的策略，且高度依赖业务的直觉（什么该优先保留）。
> 
> **总结：** 这个项目里实现一个"能跑"的 agent 不难——难的是让它在 **12 组长上下文配置**下，平均压缩 16% 后仍然能正常工作、不会裁坏当前请求、且压缩效果可解释可复现。这才是工程价值的核心。

# 5. 你怎么理解 agent 里的 memory？它本质上是在解决什么问题？

> [!NOTE]
> 我的理解是：Agent memory 本质上是在解决 **LLM 的无状态性**。
> 
> LLM 每次调用都是独立的，它在当前 prompt 之外没有任何"记忆"。但这和 agent 的工作方式冲突——agent 需要在多轮交互中：
> 
> 1. **记住已经做过的事**（任务摘要、当前进度）
> 2. **记住已经读过的信息**（文件内容、配置值）
> 3. **记住已经得出的结论**（"部署密钥是红色"、"第一行不能改"）
> 
> 如果没有 memory，以上信息只能**全部塞在聊天历史里**。这就导致了三个连锁问题：
> - prompt 无限膨胀：超预算、裁坏关键信息
> - 信息散落在历史里：每轮都需要模型重新从大量文本中"翻"出事实，重复读文件
> - 下次会话就是一片空白：中断后一切归零，只能从头来
> 
> 所以 memory 的本质是 **把信息从"线性历史"中提取出来，变成"结构化存储"，按需注入 prompt**。它解决的问题不是"让模型有记忆"——模型永远不会真正"记住"任何东西——而是**让 harness 知道该把什么信息塞回 prompt，从而让模型看起来记住了**。
> 
> 回到这个项目的三层设计：
> 
> ```
> working（当前任务摘要 + 最近文件） → 高频、小体量、随时可读
> episodic（事件笔记 + 相关记忆召回） → 按 tag/关键词跨轮检索 top-3
> durable（长期持久化笔记） → 跨会话、磁盘存储
> ```
> 
> 每一层就是在回答同一个问题的一个维度：**"现在这个时刻，模型需要知道什么才能不重读文件、不重复确认、不从头开始？"**

# 6. 如果只讲最重要的一点，你觉得 memory 设计里最大的难点是什么？

> [!NOTE]
> 我觉得最大的难点是 **"在预算约束下，从多段记忆里选出要注入的那几条"**。
> 
> 具体来说：
> 
> 1. **记忆永远不够位置放** — prompt 有 12000 字符预算，`relevant_memory` 分到 1200，按 per_note_budget 每条大概 300-400 字符，也就是最多放 **3 条笔记**
> 2. **但相关记忆可能远多于 3 条** — 一个跑了 50 轮的 agent 可能积累了 20+ 条笔记，你得从中选出最关键的 3 条
> 3. **选错了代价很大** — 没放进来的那条正好是当前需要的约束 → agent 只能重读文件 → 更长的历史 → 更挤的预算 → 恶性循环
> 
> 所以从 20 条里挑 3 条，用简单的 tag/关键词/时间排序，还要保证"挑错"的概率低到不影响任务成功率——这本质上是 **在有损压缩的前提下做检索召回**，没有完美的解决方案，只有工程上的权衡。

# 7. 在 memory 这件事上，你最关注的问题是什么：存什么、什么时候取、还是怎么避免污染？

> [!NOTE]
> 我最关注的是 **"什么时候取"**——也就是检索时机和频率。
> 
> 原因很简单：
> 
> - **[[存什么]]** 相对可控——可以用 tag 分类、去重、过期检查、canonical path 归一等规则保底
> - **怎么避免污染** 也有防御手段——freshness hash、[[tail_clip截断]]、裁剪顺序保证 current_request 不丢，结构化分层记忆（[[memory污染]]，[[幻觉循环]]）
> 
> 但 **"什么时候取"** 决定了整个系统的 risk/reward：
> 
> | 策略 | 风险 |
> |------|------|
> | **每轮都取**（当前做法） | 记忆翻出不需要的信息 → 占用预算 → 挤掉更重要的 history |
> | **只在第一轮取** | 后半程可能忘了关键事实 → 重复读文件 |
> | **让模型决定是否取** | 模型不可控，且额外多一次 tool call |
> | **基于某种信号触发**（如检测到 read_file、检测到疑惑表达） | 信号难定义、新增复杂度 |
> 
> 当前实现是"每轮都取"，这个策略很稳但也很"粗暴"。真正难的是：**有没有一个更精确的信号，能告诉你'这一轮模型需要记忆'，而不是每轮都盲塞 3 条笔记进去？**
> 
> 把该存的存好、把不该存的清掉，这两件事有规则可以写。但"什么时候喂"这个问题，规则写得再好也逃不过**不知道当前 query 和哪些记忆相关**这个根本困境——除非你用一个模型去判断另一个模型的记忆需求，而那又引入了成本、延迟和确定性下降的新问题。

# 8. 你觉得在 coding agent 场景里，memory 和普通聊天机器人的 memory 最大区别是什么？

> [!NOTE]
> 核心区别：**编码 agent 的记忆需要与执行环境保持一致性，聊天机器人不需要。**
> 
> 聊天机器人的 memory 只需要记住"用户说过什么、我喜欢什么"——信息是自包含的，存在于对话内容本身即可。
> 
> 编码 agent 的记忆，多了一个维度：**它记忆的内容，必须和实际文件系统的状态对得上。**
> 
> 具体体现：
> 
> 1. **文件摘要会过期** — `memory.py:275-279` 的 `file_freshness` 用文件内容的 SHA256 做哈希。agent 记住"README.md 第一行是引言"，但如果文件被外部修改了，这条记忆就自动失效。聊天机器人的记忆没有"外部事实校验"这一说
> 
> 2. **记忆依赖 workspace 根路径** — `canonicalize_path()` 把路径归一化到 workspace 根目录。agent 必须知道自己当前在哪个项目里，记忆才有意义。聊天的记忆不需要知道"我在哪个目录下"
> 
> 3. **记忆错了直接导致工具调用错** — agent 记错了一个 API 路径，就会调错接口、写错代码、改错文件。这个错误的**执行结果会污染后续状态**。聊天记错了，最多给个不准确的回答，不会触发一连串错误操作
> 
> 4. **预算约束更硬** — 编码任务的 prompt 里要放文件内容、工具结果、代码片段，每段都很大。记忆占的每个字符都在和这些执行上下文抢位置。聊天的记忆通常只在开头注入，竞争压力小得多
> 
> **一句话总结：聊天机器人的记忆只影响"说什么"；编码 agent 的记忆直接影响"做什么"，而且必须和执行环境的真实状态保持一致。**

8. memory 应该在什么时候检索？你认为什么时机最合理？

> [!NOTE]
> 我认为最合理的策略是**分层 + 按信号触发**，而不是每轮都取：
> 
> ### 当前做法的问题
> 
> 每轮都检索 top-3 注入，意味着：
> - 即使当前请求完全不依赖记忆（比如"把这个变量改名"），3 条笔记也占用了预算
> - 真正需要记忆的轮次（比如"前面提到的那个 API 路径是什么"），和不需要的轮次，消耗一样的预算
> 
> ### 我倾向的时机设计
> 
> **第一层——工作记忆，每轮都注入**（`task_summary` + `recent_files`）：
> 
> 体量极小（~50 字符），永远是当前任务的锚点，没有理由不带上。
> 
> **第二层——episodic notes，按信号触发**：
> 
> | 信号 | 触发时机 | 理由 |
> |------|---------|------|
> | **模型试图 `read_file`** | 检测到模型输出中的 tool call | 它要读文件 → 记忆里可能有摘要，注入后可能省掉这次调用 |
> | **上一步出现了 error/rejected** | `tool_status` 非 ok | 出错了 → 需要回顾之前的结论，避免重复踩坑 |
> | **query 与某条笔记的标签/关键词强匹配** | 匹配分超过阈值 | 大概率相关，值得注入 |
> | **用户主动问"之前"相关** | query 含"刚才""前面""之前"等词 | 明确在要求回忆 |
> 
> 无信号时，`relevant_memory` section 直接输出 `"- none"`，不占预算。
> 
> ### 这样做的收益
> 
> - 大部分简单轮次（纯编辑、纯文件操作）不注入 episodic notes，省下预算给 history
> - 真正需要回忆的轮次，反而有更多预算放完整的记忆（不止 3 条，可能 4-5 条）
> - 每轮注入的信号是可解释的：**"因为模型要读文件了，所以注入相关记忆"**，而不是"每轮都塞 3 条，但不知道为什么是这 3 条"
> 
> 当然代价是多了几个 if 分支的判断逻辑，以及"匹配阈值"需要调参——但我觉得这个复杂度比"所有轮次无差别塞 3 条"要值得。

# 26. 你会怎么设计 coding agent 的主循环？
[[pico的主循环存在的问题]]
> [!NOTE]
> 目前pico的主循环在 `runtime.py:797-967`，代码注释已经写得很清楚：
> 
> ```
> 感知 → 决策 → 行动 → 记录，然后回到感知
> ```
> 
> ```python
> while tool_steps < self.max_steps and attempts < max_attempts:
> ```
> 
> ### 1. 感知（第 808 行）
> 
> ```python
> prompt, prompt_metadata = self._build_prompt_and_metadata(user_message)
> ```
> 
> 把 prefix + memory + relevant_memory + history + current_request 组装成 prompt。**这是 memory 被注入的地方。**
> 
> ### 2. 决策（第 873 行）
> 
> ```python
> raw = self.model_client.complete(prompt, self.max_new_tokens)
> kind, payload = self.parse(raw)
> ```
> 
> 模型返回工具调用或最终答案。如果格式错误 → `kind == "retry"`，走重试分支。
> 
> ### 3. 行动（第 897-935 行）
> 
> ```python
> if kind == "tool":
>     result = self.run_tool(name, args)
>     self.record({"role": "tool", ...})   # 写回 history
>     checkpoint = self.create_checkpoint(...)  # 写检查点
>     continue  # → 回到感知（下一轮）
> ```
> 
> 工具执行结果写回历史，创建 checkpoint，`continue` 进入下一轮循环。
> 
> ### 4. 终止条件
> 
> ```python
> if kind == "final":       # 模型输出 <final>...</final>
>     return final          # 正常结束
> 
> if attempts >= max_attempts:  # 太多格式错误
>     stop_retry_limit(...)     # 重试上限终止
> 
> if tool_steps >= self.max_steps:  # 工具步数超限
>     stop_step_limit(...)           # 步数上限终止
> ```
> 
> ### 图示
> 
> ```
>                     ┌─────────────────────────┐
>                     │   组装 prompt（注入记忆）   │  ← 感知
>                     └──────────┬──────────────┘
>                                │
>                                ▼
>                     ┌─────────────────────────┐
>                     │   模型返回 tool / final   │  ← 决策
>                     └──────────┬──────────────┘
>                                │
>                     ┌──────────┴──────────┐
>                     ▼                     ▼
>           ┌─────────────────┐   ┌──────────────────┐
>           │  run_tool + 记录  │   │ 返回 final 答案    │  ← 行动
>           │  create_checkpoint│   │ promote_durable   │
>           │  continue（下一轮） │   │ return            │
>           └─────────────────┘   └──────────────────┘
> ```
> 
> **记忆系统在每轮感知阶段介入**——`_build_prompt_and_metadata` 内部调 `context_manager.build()`，其中触发 `retrieval_candidates` 将相关笔记注入 prompt。其他阶段（决策、行动）不碰记忆。

# 8. 我问的不是[[通用 ReAct]]，而是面向 coding agent 的主循环优化，你会怎么做？

> [!NOTE]
> 面向 Coding Agent 的主循环优化，是决定 Agent 能否从“只能写玩具代码”跨越到“能真正维护大型商业项目”的关键。
> 
> 正如前面提到的，传统的通用 ReAct（Thought-Action-Observation）主循环在面对代码场景时，有着致命的缺陷：**极易陷入 Debug 死循环、Token 成本雪崩、缺乏架构大局观。**
> 
> 如果让我来重构或优化一个面向 Coding Agent 的主循环，我会彻底放弃“单一同质化的 ReAct 线性循环”，转而设计一个“多层级、状态机驱动、带有悲观剪枝”的异步主循环。以下是我的具体优化策略：
> 
> ## 1. 架构升维：从单级 ReAct 演进为“双层嵌套循环”
> 
> 代码开发具有天然的层级性：先设计架构，再写函数，最后调试 Bug。因此，主循环也必须分层：
> 
> - **外层循环（宏观控制流 - State Graph）：** 负责任务拆解与契约定义。采用确定性的状态机（如 LangGraph），节点包括 `[需求解析] -> [方案设计] -> [内层代码生成] -> [CI/CD 自动化测试] -> [Code Review]`。
>     
> - **内层循环（微观执行流 - Local ReAct）：** 仅在“内层代码生成与局部调试”节点中，开启高频的、受限的 ReAct 循环。
>     
> 
> > **优化效果：** 彻底杜绝了模型在调试某个小 Bug 时，把前期的整个方案设计、甚至项目的通用配置当成垃圾信息反复吞吐，将长距离的 Token 消耗降到了最低。
> 
> ## 2. 状态资产化：用 Artifacts 机制隔离 Memory
> 
> 传统的循环把代码变化、报错、思考全部混在 `messages` 列表里，这会导致严重的 Memory 污染。
> 
> 优化手段是引入 Anthropic 先进的 **Artifacts（资产化）概念**：将代码生成区与对话推理区解耦。
> 
> - 在主循环的 State 中，开辟一个独立的 `current_code` 字段。
>     
> - 内层 ReAct 循环在 Debug 时，每一次 Action 产生的代码直接覆盖或更新 `current_code` 资产，而历史的错误代码片段**不进入**下一次的输入上下文。
>     
> - LLM 每次看到的只有：**[当前最新的全量/局部代码] + [上一次编译的错误堆栈]**。
>     
> 
> ## 3. 引入“悲观剪枝”与强制介入机制（Anti-Looping）
> 
> Coding Agent 最容易在同一个 Bug 上死磕（连续 5 次报错，由于 Thought 的惯性，它会一直生成相似的错误代码）。优化主循环必须加入**硬编码的干预策略**：
> 
> - **错误计数器（Error Counter）：** 内层循环对连续同类型的编译错误进行计数。如果连续 3 次未能解决，强制触发 **"Reflective Re-planning"（反思重规划）**。
>     
> - **换脑机制（Model Switching）：** 正常编写代码时，使用响应速度快、成本低的次旗舰模型（如 Claude 3.5 Haiku 或 GPT-4o-mini）；一旦触发错误计数器上限，主循环立刻挂起，将当前上下文路由至旗舰模型（如 Claude 3.5 Sonnet 或 Opus），利用更强的推理能力进行“破局”。
>     
> - **人类介入（Human-in-the-Loop）：** 如果强模型在 2 轮内仍未解决，主循环发出系统中断，将当前的“思考链和错误日志”可视化呈现给开发者，由人类程序员提示关键点后，Agent 再恢复循环。
>     
> 
> ## 4. 动态 RAG 检索：执行前的“精准记忆唤醒”
> 
> 正如我们之前探讨的最佳检索时机，主循环中工具调用（Action）的记忆检索需要被精心编排：
> 
> - **实现方式：** 在内层循环准备写代码（Code Generation）的前一刻，设置一个自动拦截器。拦截器提取当前要修改的函数名或类名，去向量数据库中检索项目中**所有依赖该组件、或被该组件依赖的上下游接口定义（Context Awareness）**。
>     
> - **价值：** 确保 Agent 在闭门写代码的一瞬间，其上下文窗口里精准包含了项目最新的 API 契约，避免因为“记错接口”而写出无法通过编译的代码。
>     
> 
> ## 5. 推理成本控制：投机采样与异步流式编译
> 
> 代码的验证（编译、Lint 检查、运行单元测试）是耗时大户。如果每次都等 LLM 完整输出完代码再编译，主循环的延迟（Latency）会高得无法忍受。
> 
> - **异步验证：** 优化主循环，使其支持**流式拦截（Streaming Parser）**。当大模型在 Markdown 中输出第一个 ````python ` 块并闭合时，后端异步线程立刻将其抽离并送入沙箱进行语法检查（Linting），无需等待大模型把后面的总结陈词说完。
>     
> - **Small Speculative LLM：** 利用专门针对语法格式微调过的小模型来预生成结构化的代码骨架，大模型只负责填充逻辑，拉高主循环的吞吐量。
>     
> 
> ## 总结：优化后的 Coding Agent 主循环长什么样？
> 
> 优化后的主循环不再是一个盲目的“衔尾蛇循环”，而是一个**有纪律的工业流水线**：
> 
> $$\text{输入} \longrightarrow \text{[外层] 方案设计与资产初始化} \longrightarrow \text{[内层] 精准检索上下文} \longrightarrow \text{[内层] 代码生成 (Haiku)} \longrightarrow \text{自动化测试验证}$$
> 
> - **成功：** $\longrightarrow$ 更新资产 $\longrightarrow$ 推进到下一节点。
>     
> - **失败：** $\longrightarrow$ 错误计数 $+1$ $\longrightarrow$ 资产剪枝 $\longrightarrow$ （若超限）升级模型 (Sonnet) 反思 $\longrightarrow$ 重新局部循环。
>     
> 
> 通过这种动静分离（State与Messages分离）、强弱搭配（模型分级）、硬性卡点（防止死循环）的策略，Coding Agent 的长期运行稳定性和代码产出率通常能获得质的提升。

# 8. coding agent 的主循环和普通通用 agent 的主循环，核心差别是什么？

> [!NOTE]
> Coding Agent 的主循环与通用 Agent 的主循环，核心差别不在于“流程”上，而在于**对“确定性（Determinism）”的极端追求**和**对“资产（Artifacts）”的管理方式**。
> 
> 如果把通用 Agent 比作一个“能在聊天中解决问题的智能助理”**，那么 Coding Agent 就是一个**“在沙箱内构建并验证软件系统的工程师”。
> 
> 以下是它们的核心差异对比：
> 
> ### 1. 约束目标的差异：对话 vs. 代码语义
> 
> - **通用 Agent (如聊天机器人)：** 目标是“满意度”。它的主循环关注的是语义的流畅性、上下文的连贯性。一旦用户表示“没问题了”，循环即刻结束。
>     
> - **Coding Agent：** 目标是“可编译与可运行”。它的主循环必须严格受限于编程语言的语法（Syntax）和逻辑（Semantic）约束。即使 Agent 觉得自己“说得很有道理”，如果编译器返回 `Exit Code: 1`，循环就必须判定为“未完成”并强制重试。
>     
> 
> ### 2. 状态管理的差异：消息流 vs. 资产流
> 
> 这是两者最本质的架构区别。
> 
> - **通用 Agent：** 将所有信息存入 `messages` 列表（List of Messages）。主循环的逻辑就是：`读取消息 -> 大模型生成 -> 追加消息 -> 返回`。
>     
> - **Coding Agent：** 采用“资产化（Artifact-Centric）”主循环。
>     
>     - **核心动作：** 将代码从对话流中剥离出来，存储在 `file_system` 或 `memory_workspace` 中。
>         
>     - **循环逻辑：** `读取文件状态 -> 大模型分析 -> 生成代码变更 (Diff) -> 更新文件 -> 触发编译器/测试器 -> 观察编译结果 -> 循环`。
>         
>     - **差异点：** Coding Agent 的循环主体是“文件系统”，而非“聊天记录”。
>         
> 
> ### 3. 反馈机制的差异：模糊语义反馈 vs. 硬核错误反馈
> 
> - **通用 Agent：** 反馈通常是用户的点赞/差评，或是下一轮的人类 Prompt。这种反馈是**延迟的、非标准化的**。
>     
> - **Coding Agent：** 拥有“极速闭环反馈”。
>     
>     - 编译器和测试框架是 Agent 的“眼睛”。主循环中存在一个强力的 **CI/CD 触发器**。
>         
>     - 如果代码跑不通，报错信息（Stack Trace）是唯一权威的判官，它能直接作为下一轮推理的输入（Observation），迫使 Agent 进行基于逻辑的修改，而不是基于“道歉”的修改。
>         
> 
> ### 4. 循环控制权：LLM 主导 vs. 框架主导
> 
> - **通用 Agent (ReAct 范式)：** 倾向于将控制权交给 LLM，让它自主决定何时查询、何时停止。
>     
> - **Coding Agent (Pipeline 范式)：** 倾向于**强制性的 Pipeline 编排**。
>     
>     - 它不信任模型能自主决定“什么时候该测试”，而是强制每一轮生成后必须经过 `Test` 节点。
>         
>     - 它不信任模型能自主保持架构一致，而是强制在循环头部注入 `README` 或 `Project Spec` 等全局上下文。
>         
> 
> ### 核心差异总结表
> 
> |**维度**|**通用 Agent 主循环**|**Coding Agent 主循环**|
> |---|---|---|
> |**驱动力**|语义逻辑 (Semantic)|语法与逻辑一致性 (Syntax & Logic)|
> |**主要处理对象**|文本片段 (Messages)|代码文件与抽象语法树 (Artifacts)|
> |**反馈判断者**|用户 (Human)|编译器、测试框架 (Compiler/Test Runner)|
> |**失败处理方式**|尝试解释或道歉|尝试修复 Bug 或重构 (Error-driven)|
> |**上下文管理**|窗口截断/总结|动态增删代码资产，强制覆盖陈旧逻辑|
> 
> ### 总结：为什么 Coding Agent 更复杂？
> 
> 通用 Agent 的循环是**开放式 (Open-ended)** 的，它只要不引起歧义即可；而 Coding Agent 的循环是**收敛式 (Convergent)** 的，它的每一次循环都必须向“零 Bug”这个极高标准靠近。
> 
> 对于 Coding Agent 而言，**“主循环”其实是一个“编译-测试-修复”的流水线**，大模型只是流水线上的操作员，而不是流水线的指挥官。

# 9. 你会从哪些方面专门为 coding agent 优化主循环？





## 你怎么理解 agent 里的 memory？它本质上是在解决什么问题？

> [!NOTE]
> 我理解 agent 里的 memory，不是简单“把历史聊天都存下来”，而是让 agent 在有限上下文窗口之外，维护长期有用的信息。
> 
> 它本质上解决的是：
> 
> 1. **上下文有限的问题**：当前窗口装不下所有历史。
> 
> 2. **任务连续性的问题**：agent 需要记住之前做过什么、用户偏好是什么、项目约束是什么。
> 
> 3. **减少重复探索的问题**：比如之前某个命令失败过、某个依赖有坑，下次不应该重新踩坑。
> 
> 4. **跨 session 协作的问题**：用户隔几天回来，agent 仍然能接上之前的任务状态。
> 
> 所以 memory 的本质不是存储，而是：**在有限上下文下，选择性恢复对当前任务有价值的长期状态。**

---

## 如果只讲最重要的一点，你觉得 memory 设计里最大的难点是什么？

> [!NOTE]
> 我觉得最大的难点不是“怎么存”，而是**怎么取，以及取出来以后怎么不污染当前任务**。
> 
> 因为存储本身相对容易，可以用数据库、向量库、日志系统。
> 
> 真正难的是：
> 
> 1. 当前任务到底需不需要 memory？
> 
> 2. 应该检索哪类 memory？
> 
> 3. 检索出来的信息是否过期？
> 
> 4. 它和当前用户指令、当前代码状态冲突时，谁优先？
> 
> 5. 怎么避免把旧的、错的、无关的信息塞进 prompt，影响模型判断？
> 
> 所以 memory 最大难点是：**retrieval policy \+ contamination control**，也就是检索策略和污染控制。

---

## 在 memory 这件事上，你最关注的问题是什么：存什么、什么时候取、还是怎么避免污染？

> [!NOTE]
> 三个都重要，但我最关注的是：**什么时候取，以及怎么避免污染。**
> 
> 原因是：
> 
> 1. “存什么”可以通过规则逐渐优化；
> 
> 2. 但“什么时候取”直接决定 memory 是否真正参与决策；
> 
> 3. “怎么避免污染”决定 memory 是帮助 agent，还是误导 agent。
> 
> 尤其在 agent 场景里，memory 不是越多越好。取太少，agent 像失忆；取太多，agent 会被旧信息和噪声干扰。
> 
> 所以我的优先级是：
> 
> 4. **什么时候检索**
> 
> 5. **检索什么**
> 
> 6. **检索结果如何校验**
> 
> 7. **怎么注入 prompt**
> 
> 8. **哪些内容值得写入 memory**

---

## 你觉得在 coding agent 场景里，memory 和普通聊天机器人的 memory 最大区别是什么？

> [!NOTE]
> 最大区别是：**coding agent 的 memory 必须和项目状态绑定，而且必须可验证、可失效。**
> 
> 普通聊天机器人的 memory 更多是用户偏好，例如：
> 
> 1. 用户喜欢中文回答；
> 
> 2. 用户喜欢简洁风格；
> 
> 3. 用户的长期兴趣；
> 
> 4. 用户的称呼偏好。
> 
> 但 coding agent 的 memory 更像项目工作记忆，例如：
> 
> 1. 这个 repo 用 pnpm，不用 npm；
> 
> 2. 测试命令是 `pnpm test`；
> 
> 3. 某个模块不能直接改，因为有兼容性约束；
> 
> 4. 这个项目使用某种架构模式；
> 
> 5. 某个 bug 之前的根因是什么；
> 
> 6. 某个失败尝试已经被验证过，不要重复做。
> 
> coding memory 的风险也更高。因为普通聊天 memory 错了，可能只是回答风格不准；coding memory 错了，可能会导致 agent 改错代码、跑错命令、破坏项目约束。
> 
> 所以 coding agent 的 memory 必须带上：
> 
> 1. repo；
> 
> 2. branch；
> 
> 3. 文件路径；
> 
> 4. 时间；
> 
> 5. 来源；
> 
> 6. 置信度；
> 
> 7. 是否仍然有效。

---

## memory 应该在什么时候检索？你认为什么时机最合理？

我认为最合理的时机是：**agent 理解当前任务之后，正式制定计划之前。**

也就是说，不是一收到用户消息就立刻检索，而是先做一层 task understanding：

1. 用户要做什么？

2. 涉及哪个项目？

3. 涉及什么文件或模块？

4. 是 bug fix、feature、refactor，还是测试问题？

5. 是否有跨 session 信息需求？

然后再基于这个理解去检索 memory。

除此之外，执行过程中也需要按需检索，比如：

1. 用户说“还是上次那个问题”；

2. agent 遇到测试失败；

3. agent 不确定项目约定；

4. agent 进入某个陌生模块；

5. 当前观察和已有信息冲突；

6. 需要查历史失败尝试或历史决策。

所以我会设计成：**开始前一次主动检索，执行中按需触发检索。**

---

## 为什么你认为那个时机是最好的？

因为这个时机平衡了两个问题：太早检索不准，太晚检索没用。

如果太早检索，agent 还没有理解任务，只能用用户原始输入去搜，很容易召回无关 memory。

如果太晚检索，agent 已经制定了计划甚至开始执行，memory 只能补救，很难影响核心决策。

而“理解任务之后、制定计划之前”这个时机最好，因为：

1. agent 已经知道当前任务意图；

2. 可以生成更准确的检索 query；

3. 可以用 repo、文件、任务类型做过滤；

4. memory 能真正影响 plan；

5. 可以避免 agent 先入为主地走错方向。

所以这个时机既有足够上下文，又还没有锁定执行路径。

---

## memory 检索这件事，怎么真正落到工程实践里？

工程上我不会让模型随便“想查就查”，而是会设计一个明确的 memory retrieval pipeline。

大概流程是：

1. **Task Parser**
先解析当前任务，提取用户意图、项目、文件、任务类型、约束。

2. **Memory Router**
判断这个任务是否需要 memory。比如简单问答可能不需要，跨 session coding 任务通常需要。

3. **Query Builder**
生成结构化 query，不只用自然语言，还要带 metadata，比如 repo、branch、file path、task type。

4. **Hybrid Retrieval**
同时用语义检索和关键词检索。语义检索找相似经验，关键词检索找精确文件名、命令、错误信息。

5. **Metadata Filtering**
过滤掉不同项目、不同分支、过期或低置信度的 memory。

6. **Reranking**
根据相关性、新鲜度、可靠性、来源证据重新排序。

7. **Conflict Check**
检查 memory 是否和当前工具观察、当前代码状态冲突。

8. **Prompt Injection**
只把少量高价值 memory 放进 prompt，并标明来源、时间、适用范围和置信度。

---

## 如果让你设计一个可落地的 memory 检索流程，你会怎么做？

我会设计成这几个步骤：

第一步，先判断是否需要 memory。

比如：

1. 用户提到“之前”“上次”“继续”；

2. 当前任务涉及已有项目；

3. 任务需要遵守用户偏好；

4. 任务涉及历史 bug、历史决策、历史失败尝试；

5. 当前上下文信息不足。

第二步，生成检索 query。

query 不应该只是用户原话，而应该结构化，例如：

```Plain Text
repo: payment-service
branch: feature/refactor-checkout
task_type: bug_fix
files: checkout.ts, paymentClient.ts
keywords: timeout, retry, integration test
```

第三步，做 hybrid search。

同时使用：

1. 向量检索；

2. 关键词检索；

3. metadata 过滤；

4. 时间过滤；

5. 项目过滤。

第四步，对结果 rerank。

排序标准包括：

1. 和当前任务的相关性；

2. 是否同一个 repo；

3. 是否同一个 branch；

4. 是否有证据来源；

5. 是否最近；

6. 是否被验证过；

7. 是否和当前代码状态冲突。

第五步，压缩并注入 prompt。

只保留最有用的几条 memory，并组织成：

```Plain Text
Relevant Memory:
- [Project Constraint] This repo uses pnpm, not npm. Source: previous successful test run. Scope: repo-level.
- [Past Failure] Running npm install caused lockfile conflict. Source: previous agent run. Confidence: high.
- [User Preference] User prefers minimal changes and tests before refactor. Scope: user-level.
```

第六步，执行中动态更新。

如果执行中发现 memory 过期，要标记失效；如果发现新的长期有用信息，要进入写入候选区，而不是马上无脑写入。

---

## 你会如何决定“这条信息值得进 memory”还是“只留在当前上下文里”？

我会用几个标准判断。

值得进长期 memory 的信息通常满足：

1. **跨 session 仍然有用**
不是只对当前这一轮有用。

2. **未来复用价值高**
以后遇到类似任务还能用。

3. **重新发现成本高**
如果忘了，下次要花很多时间重新探索。

4. **已经被验证过**
不是模型猜测，而是有工具结果、用户确认或代码证据。

5. **作用范围清楚**
知道它适用于哪个用户、哪个项目、哪个 repo、哪个文件或哪类任务。

6. **相对稳定**
不会很快过期；如果可能过期，就要带有效期或版本信息。

比如这些值得进 memory：

1. 用户偏好：“用户希望 coding 回答先给方案，再改代码。”

2. 项目约束：“这个 repo 使用 pnpm。”

3. 环境信息：“集成测试需要本地 Postgres。”

4. 历史教训：“之前直接升级依赖导致兼容性问题。”

5. 架构决策：“认证逻辑集中在 auth middleware，不要在 controller 里重复实现。”

不值得进长期 memory 的包括：

1. 临时中间推理；

2. 一次性任务细节；

3. 没验证的猜测；

4. 工具输出里的大量噪声；

5. 已经过期的信息；

6. 和当前事实冲突的信息；

7. 用户只是随口说但没有长期价值的内容。

我的原则是：**memory 应该存稳定、可复用、已验证、作用范围清楚的信息，而不是存所有历史。**

---

## 检索出来的 memory 怎么重新组织进 prompt，才不会干扰当前任务？

我会遵守几个原则。

第一，memory 要单独成区，不要混在用户指令里。

比如放成：

```Plain Text
Relevant Memory
```

而不是直接拼到用户消息后面。

第二，每条 memory 要短，不要塞原始长文本。

应该是压缩后的结构化信息，例如：

```Plain Text
- Type: Project Constraint
  Content: This repo uses pnpm instead of npm.
  Scope: repo-level
  Source: previous successful setup
  Confidence: high
```

第三，要标注作用范围。

比如：

1. user\-level；

2. repo\-level；

3. branch\-level；

4. file\-level；

5. task\-level。

这样模型知道这条 memory 应该在哪里生效。

第四，要标注时间和来源。

因为 coding memory 很容易过期。没有来源和时间的 memory 不应该拥有很高优先级。

第五，要明确优先级。

我会在 prompt 里写清楚：

```Plain Text
Priority:
1. Current user instruction
2. Current tool observations and repository state
3. Current conversation context
4. Retrieved memory
```

也就是说，memory 只是参考信息，不能覆盖当前用户指令和当前真实代码状态。

第六，要控制数量。

不要把几十条 memory 都塞进去。宁愿只放 3 到 5 条高置信 memory，也不要放一堆可能相关的噪声。

所以我会把 memory 注入 prompt 的原则总结成：

**独立区域、短摘要、带来源、带范围、带置信度、低于当前事实优先级。**

---

## 总结

如果总结成一句面试回答，我会说：

Agent memory 的核心不是让模型记住一切，而是在有限上下文下，选择性保存和检索对当前任务有长期价值的信息。真正难点不在存储，而在检索时机、相关性判断、过期信息处理和污染控制。尤其在 coding agent 里，memory 必须和 repo、branch、文件路径、工具结果绑定，并且始终低于当前用户指令和当前代码状态。

## “agent 是脑子，sandbox 是手”，如果有多个 sandbox 环境，它们应该如何被统一组织和调度？

我会把这个问题理解成：agent 负责决策，sandbox 负责执行，但当 sandbox 不止一个时，中间必须有一个统一的 **orchestration layer**，也就是调度和编排层。

不能让 agent 直接面对多个 sandbox，否则会出现几个问题：

1. agent 不知道每个 sandbox 的状态；

2. agent 不知道哪个 sandbox 有权限；

3. agent 不知道哪个 sandbox 里有什么文件和依赖；

4. agent 可能把任务发到错误环境；

5. 多个 sandbox 的执行结果难以合并。

所以我会设计成三层：

```Plain Text
Agent / Planner
      ↓
Sandbox Orchestrator
      ↓
Sandbox Pool
```

第一层是 **Agent / Planner**。
它只负责理解任务、拆分步骤、决定要执行什么动作，比如运行测试、修改代码、访问网页、执行脚本。

第二层是 **Sandbox Orchestrator**。
这是关键。它负责把 agent 的意图转换成具体的执行计划，比如：

1. 选择哪个 sandbox；

2. 创建还是复用 sandbox；

3. 分配权限；

4. 注入必要文件；

5. 执行命令；

6. 收集日志和产物；

7. 把结果结构化返回给 agent。

第三层是 **Sandbox Pool**。
这里可以有多个不同类型的 sandbox，例如：

1. 代码执行 sandbox；

2. 浏览器 sandbox；

3. repo workspace sandbox；

4. 数据分析 sandbox；

5. 测试环境 sandbox；

6. 隔离的高风险操作 sandbox。

所以多个 sandbox 的统一组织方式，不是让 agent 一个个记住它们，而是把它们抽象成一组能力：

```Plain Text
Capability:
- run_code
- edit_repo
- run_tests
- browse_web
- analyze_file
- call_internal_tool
```

agent 只表达“我要什么能力”，orchestrator 决定“用哪个 sandbox 来完成”。

---

## 如果不讨论模型本身、不讨论 sandbox 本身，只讨论工程组织逻辑，你会怎么让这两个部分真正协作起来？

我会让 agent 和 sandbox 之间通过一个标准化协议协作，而不是直接耦合。

核心是：**agent 输出意图，orchestrator 负责执行，sandbox 返回观察结果，agent 再基于观察结果继续决策。**

流程可以是：

```Plain Text
User Task
  ↓
Agent 理解任务
  ↓
Agent 生成 action request
  ↓
Orchestrator 选择 sandbox
  ↓
Sandbox 执行
  ↓
返回 observation / artifact / error
  ↓
Agent 更新计划
```

这里最重要的是把接口设计清楚。

agent 不应该直接说：

```Plain Text
去 sandbox-3 里面跑这个命令
```

而应该说：

```Plain Text
我需要在当前 repo 环境里运行单元测试
```

然后 orchestrator 判断：

1. 当前有没有已经绑定这个 repo 的 sandbox；

2. 这个 sandbox 是否空闲；

3. 是否有运行测试的权限；

4. 依赖是否已经安装；

5. 是否需要创建新 sandbox；

6. 是否需要复用之前的 workspace。

也就是说，agent 负责“要做什么”，orchestrator 负责“在哪里做、怎么做、用什么权限做”。

工程上我会把协作拆成几个模块：

1. **Task Manager**
管理用户任务、子任务、执行状态。

2. **Planner / Agent Runtime**
生成下一步 action，但不直接操作底层环境。

3. **Tool Gateway**
把 agent 的 action request 转成标准工具调用。

4. **Sandbox Scheduler**
选择、创建、复用、销毁 sandbox。

5. **State Store**
记录任务状态、sandbox 状态、文件状态、执行日志。

6. **Artifact Store**
保存执行结果，比如代码 diff、测试报告、生成文件、日志摘要。

7. **Policy Engine**
控制权限，比如是否允许联网、是否允许访问 secrets、是否允许写文件。

8. **Observation Formatter**
把 sandbox 返回的大量日志压缩成 agent 能理解的结构化结果。

这样 agent 和 sandbox 就不会互相侵入。agent 不需要知道 sandbox 的底层实现，sandbox 也不需要理解 agent 的完整推理过程。

---

## 多个 sandbox 并存时，怎么管理状态、权限和任务分发？

我会分三块讲：状态管理、权限管理、任务分发。

### 第一，状态管理

多个 sandbox 并存时，最容易乱的是状态。

我会把状态分成几类：

1. **Task State**
当前任务做到了哪一步，哪些子任务完成了，哪些失败了。

2. **Sandbox State**
每个 sandbox 当前是 idle、running、failed、expired，还是 locked。

3. **Workspace State**
这个 sandbox 绑定的是哪个 repo、哪个 branch、哪些文件、哪些依赖。

4. **Artifact State**
每次执行产生了什么结果，比如日志、diff、测试报告、临时文件。

5. **Memory / Metadata State**
哪些信息值得长期保存，比如这个项目的测试命令、依赖约束、失败经验。

关键原则是：**不要把重要状态只留在 sandbox 里。**

因为 sandbox 可能被销毁、重启、超时或污染。真正重要的状态应该沉淀到外部：

```Plain Text
State Store + Artifact Store
```

sandbox 只是执行现场，不是唯一事实来源。

我会让每个 sandbox 都有明确的 metadata，例如：

```Plain Text
sandbox_id
task_id
repo_id
branch
capabilities
permission_scope
status
created_at
last_used_at
owner_agent_session
```

这样 orchestrator 才知道哪个 sandbox 可以复用，哪个必须销毁，哪个不能混用。

---

### 第二，权限管理

多个 sandbox 并存时，权限不能靠 agent 自觉，而要由工程系统控制。

我会用几个原则：

1. **最小权限原则**
一个 sandbox 只拿完成当前任务需要的权限。

2. **按任务发放权限**
权限和 task\_id 绑定，而不是给某个 sandbox 永久开放。

3. **能力级权限控制**
比如：

    - 是否能联网；

    - 是否能写文件；

    - 是否能访问 repo；

    - 是否能读取 secrets；

    - 是否能调用外部 API；

    - 是否能执行危险命令。

4. **敏感操作需要审批或额外校验**
比如删除文件、修改生产配置、访问密钥、调用付费 API。

5. **权限可回收**
任务结束后，sandbox 的 token、secret、临时凭证都应该失效。

6. **全量审计日志**
所有命令、文件修改、外部访问、权限申请都要记录，方便回溯。

所以权限管理不应该是：

```Plain Text
agent 想做什么就做什么
```

而应该是：

```Plain Text
agent 提出 action
orchestrator 检查 policy
policy 允许后才给 sandbox 执行
```

---

### 第三，任务分发

多个 sandbox 的任务分发，我会按任务类型、资源需求和隔离级别来调度。

比如：

1. 简单代码运行 → 分配轻量 code sandbox；

2. repo 修改 → 分配绑定 repo 的 workspace sandbox；

3. 长时间测试 → 分配可持久化的 test sandbox；

4. 高风险命令 → 分配一次性隔离 sandbox；

5. 多文件分析 → 可以并行分发到多个只读 sandbox；

6. 需要共享上下文的任务 → 复用同一个 project sandbox。

任务分发流程可以是：

```Plain Text
1. 解析任务类型
2. 判断所需 capability
3. 检查是否已有可复用 sandbox
4. 如果没有，创建新 sandbox
5. 注入必要上下文和文件
6. 执行任务
7. 收集结果
8. 更新 task state
9. 决定复用、冻结或销毁 sandbox
```

如果任务可以拆分，还可以做 DAG 调度：

```Plain Text
Analyze repo
   ↓
Run tests ──→ Inspect failure
   ↓              ↓
Modify code → Re-run tests
   ↓
Generate final diff
```

每个节点都可以交给不同 sandbox，但 orchestrator 要负责依赖关系和结果合并。

## 总结回答

如果总结成一段面试回答，我会说：

当 agent 是脑子、sandbox 是手时，多个 sandbox 不能由 agent 直接管理，而应该通过一个统一的 sandbox orchestration layer 来组织。agent 只表达任务意图和所需能力，orchestrator 负责任务拆分、sandbox 选择、权限控制、状态管理、执行调度和结果汇总。

多个 sandbox 并存时，我会把它们抽象成 capability pool，而不是具体机器。状态上，用外部 state store 和 artifact store 管理任务状态、workspace 状态和执行产物；权限上，用 policy engine 做最小权限、临时授权和审计；任务分发上，根据任务类型、资源需求、隔离级别和上下文依赖选择创建、复用或销毁 sandbox。

核心原则是：**agent 负责决策，sandbox 负责执行，orchestrator 负责把决策安全、可控、可追踪地落到执行环境里。**

下面按 18–24 分点回答。

### 为什么模型的注意力是有限的？

模型的“注意力有限”主要有几层原因：

1. **上下文窗口有限**
模型一次只能看到固定长度的 token。超出窗口的内容不会直接参与当前推理。

2. **计算资源有限**
Transformer 的自注意力会让每个 token 和其他 token 建立关系，长上下文会显著增加计算和显存成本。Transformer 架构正是基于 attention 机制处理序列信息，但它不是无限容量的记忆系统。\([NeurIPS Papers](https://papers.neurips.cc/paper/7181-attention-is-all-you-need.pdf?utm_source=chatgpt.com)\)

3. **注意力不是“完美检索”**
attention 只是给不同 token 分配权重，不等于模型能稳定、精确地从任意位置取回任意信息。

4. **模型层数、注意力头数有限**
每一层、每个 head 都只能表达有限的关系。上下文越长，竞争同一注意力资源的信息越多。

5. **训练分布限制**
即使模型支持长上下文，它也未必在训练中充分学会“长距离精确检索”和“跨很远位置稳定遵从指令”。

---

### 为什么长上下文里模型更容易记住前面的内容和最后的内容，而容易遗忘中间内容？

这就是常说的 **“lost in the middle / 中间遗忘”** 现象。研究发现，模型在长上下文中使用信息并不均匀：相关信息放在开头或结尾时，表现通常更好；放在中间时，性能会明显下降。\([arXiv](https://arxiv.org/abs/2307.03172?utm_source=chatgpt.com)\)

原因包括：

1. **开头有“任务设定优势”**
系统提示、角色设定、全局目标通常放在最前面，模型容易把它们当成高层约束。

2. **结尾有“近因优势”**
生成答案时，最近出现的内容离输出位置最近，更容易影响下一步 token 预测。

3. **中间内容被两端挤压**
中间既没有开头的全局设定地位，也没有结尾的近距离优势，容易在长上下文中被大量细节稀释。

4. **位置编码和长距离依赖问题**
模型虽然能表示位置，但长距离关系并不一定稳定，尤其当上下文远超常见训练长度时。

5. **注意力竞争**
长上下文里有很多相似、重复、过期或冲突的信息，中间的关键内容可能拿不到足够高的注意力权重。

---

### 这种“中间遗忘”现象，对 coding agent 的设计会带来什么影响？

对 coding agent 影响很大，因为 coding agent 经常有长历史：需求、错误日志、文件内容、修改记录、测试输出、用户纠正等。

主要影响：

1. **用户当前指令可能被旧任务状态污染**
agent 可能继续执行之前的目标，而不是最新目标。

2. **中间的关键约束容易丢失**
例如用户早先说“不要改 API”，后来大量日志和代码塞进上下文，中间这条约束可能被忽略。

3. **容易重复旧错误**
如果“之前已经试过且失败”的信息在中间，agent 可能再次尝试同样方案。

4. **代码修改容易失去全局一致性**
agent 只记住最近的报错，却忘了前面设计原则，导致补丁式修复。

5. **长任务需要显式状态管理**
coding agent 不能只依赖自然语言上下文，需要结构化 task state、decision log、current instruction、constraints、done/undone 列表。

---

### 在 memory 压缩或上下文裁剪的场景下，怎么增强 agent 对用户指令的遵从性？

可以从 memory 结构上增强，而不是只做普通摘要。

1. **把用户指令单独抽取成最高优先级块**
不要让用户指令混在历史摘要里。应单独维护：

    - 当前目标

    - 用户明确要求

    - 用户禁止事项

    - 输出格式要求

    - 最近一次用户纠正

2. **压缩时保留“约束”，而不是只保留“事实”**
很多摘要只保留发生了什么，却丢掉“不能做什么”“必须怎么做”。

3. **区分历史状态和当前命令**
历史状态只是参考，当前用户指令才是执行目标。

4. **对冲突信息做显式标记**
例如：

    - 旧要求：使用 Python

    - 新要求：改用 TypeScript

    - 决策：当前指令覆盖旧要求

5. **压缩后做 instruction audit**
在执行前让 agent 自检：

    - 当前用户到底要什么？

    - 有哪些禁止操作？

    - 历史信息里有没有与当前指令冲突的内容？

    - 本轮输出必须满足哪些格式？

---

### 如果上下文必须压缩，你怎么确保用户当前指令不会被历史信息淹没？

核心原则：**当前指令不压缩、不降权、不混入历史摘要。**

具体做法：

1. **当前用户消息原文保留在上下文末尾**
不只保留摘要，尽量保留原文。

2. **把当前请求复制到“Active Instruction”区**
例如：

```Plain Text
Active User Request:
- 用户当前要求：……
- 本轮必须回答：……
- 不要执行：……
```

1. **历史内容只能作为 supporting context**
历史信息不能自动覆盖当前请求。

2. **摘要分层**
推荐结构：

    - Non\-negotiable instructions

    - Current user request

    - Relevant task state

    - Historical decisions

    - Discarded / outdated context

3. **裁剪优先级固定化**
保留顺序应是：

4. 当前用户指令 \&gt; 系统/开发者约束 \&gt; 用户长期偏好 \&gt; 当前任务状态 \&gt; 最近错误 \&gt; 历史摘要 \&gt; 原始日志

---

### 你会怎么设计 prompt 结构，保证“当前请求优先级最高”？

可以设计成这种结构：

```Plain Text
[System / Developer Constraints]
你必须遵守的全局规则。

[Current User Request - Highest Runtime Priority]
这是当前用户的最新请求：
<原文或忠实重述>

执行规则：
1. 当前请求优先于历史任务状态。
2. 如果历史信息与当前请求冲突，以当前请求为准。
3. 不要继续执行旧目标，除非当前请求明确要求。
4. 回答前先确认本轮真正要完成的任务。

[Current Instruction Checklist]
- 用户要我做什么：
- 用户不希望我做什么：
- 输出格式要求：
- 需要引用的当前文件/代码/错误：
- 不确定但不能擅自假设的点：

[Relevant Task State]
只放和当前请求直接相关的历史状态。

[Historical Context - Lower Priority]
旧决策、旧尝试、旧错误、旧文件摘要。

[Execution]
根据 Current User Request 生成答案或执行修改。
```

这个结构的重点是：**当前请求既出现在上下文末尾，又被显式提升为 runtime priority。**

---

### 在 coding agent 里，如何平衡“保留历史任务状态”和“严格执行当前用户指令”？

可以按以下原则设计：

1. **分离“状态记忆”和“执行指令”**

    - 状态记忆：项目结构、已改文件、测试结果、历史决策。

    - 执行指令：本轮用户到底要 agent 做什么。

2. 状态记忆只能辅助执行，不能替代当前指令。

3. **当前用户指令永远进入 active task**

4. 每轮都重建：

```Plain Text
active_task = latest_user_request
```

1. 而不是默认延续上一轮任务。

2. **历史状态只保留可验证事实**

3. 好的历史状态：

    - `src/auth\.ts 已修改`

    - `npm test 失败，错误是 X`

    - `用户要求不要改 public API`

4. 差的历史状态：

    - `继续优化代码`

    - `用户可能想要重构`

    - `之前的方向应该继续`

5. **对旧目标设置过期机制**

6. 旧任务不能无限期有效。可以标记：

    - active

    - completed

    - superseded

    - blocked

    - abandoned

7. **冲突时明确采用最新用户指令**

8. 例如：

```Plain Text
历史状态：之前计划重构整个模块。
当前指令：只修复一个 bug，不做重构。
执行策略：只做最小 bug fix。
```

1. **每次执行前做 scope check**

2. coding agent 应先判断：

    - 本轮是否需要改代码？

    - 允许改哪些文件？

    - 是否允许重构？

    - 是否需要运行测试？

    - 用户有没有禁止某些操作？

3. **把用户约束放进 patch planning**

4. 不是先写代码再检查，而是在生成 patch 前就约束：

```Plain Text
Patch Plan:
- Must satisfy current request
- Must not touch unrelated files
- Must preserve public API
- Must avoid previous failed approach
```

1. **长期任务用 task ledger，而不是完整聊天历史**

2. 推荐维护结构化 ledger：

```Plain Text
Task Ledger:
- Current objective
- User constraints
- Files touched
- Decisions made
- Failed attempts
- Open questions
- Next action
```

1. **用户纠正优先级高于 agent 自己的计划**

2. 用户说“不是这个意思”“不要这样改”“只回答问题”时，应立即覆盖之前的 agent 计划。

3. **输出时显式对齐当前请求**

回答开头可以先锚定：

```Plain Text
按你当前的要求，我只处理 X，不继续之前的 Y。
```

一句话总结：
**coding agent 的 memory 应该服务于当前指令，而不是让当前指令服从历史上下文。**



下面这 7 个问题，我按“面试可直接回答”的口径给你整理。表述会尽量贴近 `pico\-v3` 这种本地 coding agent 的设计，不乱编成通用 agent 套话。



## **25\. 如果让你设计一个本地代码 agent，用户给你一个仓库，并说“登录接口偶发 500”，你会怎么设计这个 agent 的主流程？**



我会把这个流程设计成一个“先收敛现场，再定位，再验证，再交付”的闭环，而不是一上来就改代码。

第一步是仓库建模。agent 进入仓库后，先识别项目类型、启动方式、测试方式、日志位置、登录接口相关模块，把这些信息整理成当前任务上下文。  

第二步是问题澄清与目标固化。把“登录接口偶发 500”翻译成可执行目标，比如接口路径是什么、预期行为是什么、是否有已知复现条件、最终要产出修复代码还是先给根因分析。  

第三步是证据收集。优先读登录链路相关代码、异常处理、数据库访问、鉴权中间件、配置读取、外部依赖调用，同时查测试、日志、最近变更。  

第四步是构造假设。比如是空值分支、并发竞争、缓存击穿、数据库连接抖动、上游超时、异常没被正确兜底。  

第五步是验证假设。通过读代码、跑测试、定向构造输入、必要时加临时日志或最小复现脚本，去确认到底是哪一类问题。  

第六步才是修复。修复时不仅改出错点，还要补齐防御性处理，比如超时、重试边界、错误码转换、空值保护、日志字段。  

第七步是回归验证。重新跑相关测试、接口测试，确认没有引入新的行为回退。  

第八步是交付结果。输出“根因、改动、验证证据、剩余风险”四件事。

如果用一句话概括，这个 agent 的主流程不是“问一句答一句”，而是“围绕一个代码问题持续收集证据并收敛到可验证修复”。

## **26\. 针对这种 coding agent，你会设计哪些核心工具？**

我会把工具分成六类：

1. 仓库理解工具  

比如列目录、读文件、全文检索、按符号找定义、查调用链。

2. 执行与验证工具  

比如运行测试、运行单个用例、启动服务、发接口请求、查看退出码。

3. 诊断工具  

比如查看日志、抓 traceback、查看最近改动、对比前后版本、查配置和环境变量。

4. 编辑工具  

比如精确改文件、生成 patch、批量替换、插入测试。

5. 上下文治理工具  

比如计划管理、todo、会话历史、文件摘要、checkpoint、恢复执行。

6. 安全与权限工具  

比如沙盒执行、危险命令拦截、用户确认、工具白名单。

如果只看 `pico\-v3` 这种方向，真正关键的不只是“能读写文件”，而是“工具调用结果能进入后续上下文，并且能被治理”。



---



**27\. 这些工具为什么是必要的？哪些是最小闭环？**



这些工具必要，是因为 coding agent 不是纯推理任务，而是“读代码 \- 做假设 \- 执行验证 \- 修改 \- 再验证”的工程闭环。

没有仓库理解工具，它不知道代码在哪。  

没有执行验证工具，它只能猜，不能证伪。  

没有编辑工具，它只能分析，不能交付结果。  

没有上下文治理，它几轮之后就会忘。  

没有安全控制，本地 agent 很容易越权或误操作。

最小闭环我认为只有五个工具就够了：

1. `list/read`：看仓库和读文件  

2. `search`：定位接口、异常、调用链  

3. `edit/apply\_patch`：修改代码  

4. `run\_test\_or\_command`：验证修复是否成立  

5. `history/checkpoint`：把已发现的证据和结论留住

这五个加起来，才构成最小可工作的 coding agent。  

如果连测试执行都没有，那它更像“代码问答器”，不是“代码代理”。

---



## **28\. 你会怎么设计 coding agent 的主循环？**

我会把它设计成面向代码问题收敛的循环，而不是面向对话的循环。

一个比较好的主循环是：

1. 解析当前目标  

把用户问题转成任务目标、验收条件、约束条件。

2. 建立当前工作上下文  

载入仓库结构、最近文件、历史结论、相关记忆。

3. 选择下一步最小动作  

不是一次做很多，而是挑当前信息增益最大的一个动作，比如先搜登录接口定义，而不是直接改代码。

4. 执行动作  

调用读文件、搜索、测试、编辑等工具。

5. 记录结果  

把结果写回 history、todo、文件摘要、任务摘要。

6. 判断是否收敛  

如果已经定位根因，就进入修复；如果修复后验证通过，就准备交付；如果证据不足，就继续探索。

7. 做上下文压缩  

把长日志、长文件、长输出压成可继续使用的摘要，避免上下文爆炸。

8. 继续下一轮，直到满足停止条件  

比如问题复现并定位、修复并验证、或明确受阻并给出阻塞点。

这个循环的重点是：每一步都围绕“让问题更接近被验证地解决”。

## **29\. 我问的不是通用 ReAct，而是面向 coding agent 的主循环优化，你会怎么做？**

如果是面向 coding agent，我会做六个专门优化。

1. 先读后写  

默认先搜、先读、先建模，禁止模型一上来盲改。代码问题里，误改成本很高。

2. 以“证据”驱动，而不是以“动作数量”驱动  

普通 ReAct 容易变成一直调用工具。coding agent 更应该问：我现在缺什么证据，哪个动作最能缩小不确定性？

3. 把“验证”前置成主循环的核心节点  

不是最后补测一下，而是每形成一个关键假设就要设计验证动作。

4. 引入工作记忆和文件摘要  

代码仓库太大，不能每轮都塞全量上下文。要保留任务摘要、最近文件、关键结论、文件级摘要。

5. 对失败重试做结构化处理  

比如模型输出格式错了、工具失败了、测试没跑起来，这些都不能简单重来，而要把错误原因写回上下文，让下一轮知道为什么失败。

6. 把停止条件工程化  

比如“已拿到稳定复现路径”“已验证根因”“补丁通过测试”“剩余风险已说明”，这样 agent 不会无休止探索。

所以我理解的优化，不是把 ReAct 改个名字，而是把它变成适合软件调试和修复的执行系统。

## **30\. coding agent 的主循环和普通通用 agent 的主循环，核心差别是什么？**

我觉得核心差别只有一句话：

普通 agent 更关注“回答是否合理”，coding agent 更关注“改动是否可验证地正确”。

再展开一点，有四个差别：

1. 环境不同  

普通 agent 主要处理文本世界，coding agent 处理的是仓库、命令、测试、日志、依赖、运行时。

2. 成功标准不同  

普通 agent 很多时候回答得像样就行；coding agent 必须落到代码改动和验证结果。

3. 上下文结构不同  

普通 agent 主要靠对话历史；coding agent 还要维护文件状态、测试结果、补丁、工作记忆、checkpoint。

4. 风险模型不同  

普通 agent 答错了通常是信息错误；coding agent 改错了可能直接引入 bug、删数据、破坏环境，所以要有更强的权限和执行边界。

所以 coding agent 本质上不是“大模型加几个工具”，而是“带工程闭环的受控执行体”。

## **31\. 你会从哪些方面专门为 coding agent 优化主循环？分点回答**

> [!NOTE]
> 可以分 8 点回答，这样最完整也最好面试发挥：
> 
> 1. 目标建模优化  
> 
> 把用户一句话问题，转成任务目标、验收标准、风险边界、输出格式。
> 
> 2. 仓库建模优化  
> 
> 先识别项目类型、目录结构、入口文件、测试框架、依赖关系，避免无头苍蝇式搜索。
> 
> 3. 上下文治理优化  
> 
> 维护任务摘要、最近文件、关键证据、文件摘要，控制 prompt 预算，避免长上下文失焦。
> 
> 4. 动作选择优化  
> 
> 优先选择信息增益最高、风险最低的动作，比如先读登录链路，再跑最小验证，不盲目全量测试。
> 
> 5. 验证闭环优化  
> 
> 要求每个重要假设都要有验证动作，每个修复都要有回归验证，而不是只做代码层面的“看起来对”。
> 
> 6. 编辑策略优化  
> 
> 优先最小 patch、局部修改、补测试，不做大面积重构，降低引入新问题的概率。
> 
> 7. 失败恢复优化  
> 
> 把工具失败、解析失败、测试失败、环境失败都结构化写入历史，让下一轮知道“为什么上轮没成功”。
> 
> 8. 安全控制优化  
> 
> 沙盒执行、权限分级、危险命令确认、可恢复 checkpoint，这些对本地 coding agent 特别重要。
> 
> 如果面试官追问“你觉得最重要的是哪一个”，我会回答：  
> 
> 最重要的是“上下文治理 \+ 验证闭环”。因为 coding agent 真正容易失败的地方，不是不会改代码，而是忘了关键约束，或者改完没有可靠验证。
