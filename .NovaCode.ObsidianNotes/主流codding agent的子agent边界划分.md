面试高分表达（体现架构思维）

如果你在面试中向面试官阐述这个设计，可以这样总结：

> “在设计主流 Coding Agent 的多 Agent 协同（Multi-Agent Collaboration）时，我们划分边界的核心思想是 **职责单一化** 和 **状态显式化** 。
> 
> 1. 我们通过 **限制工具集（Tool Set）** 来强行框定子 Agent 的物理边界（如 Tester 只能跑测试不能改代码）。
>     
> 2. 我们通过 **图结构（如 LangGraph）的状态转移** 来框定逻辑边界。
>     
> 
> 这种划分带来的最大好处，是**控制了每个子 Agent 的上下文窗口（Context Window）**。Coder 只需要看局部的 200 行代码和改动要求，而不需要看整个项目的几万行代码。这不仅**极大地降低了 Token 的消耗**，避免了无谓的成本，更重要的是**把大模型的注意力集中在极小的范围内，从而将代码生成的准确率（如 SWE-bench 解决率）提升数倍**。”


在主流的 Coding Agent（如 Devin、SWE-agent、OpenHands，以及企业级落地的多 Agent 编程框架）中，通常不会让一个大模型“包揽一切”。因为编写代码是一个集**需求分析、架构设计、代码编写、编译报错、跑单测、Debug**于一体的复杂长链路任务，单模型极易在长链路中迷失。

为了解决这个问题，主流架构普遍采用 **“分治法”（Divide and Conquer）**，将任务拆解给不同的**子 Agent（Sub-agents）。**

这些子 Agent 的边界划分非常精妙，通常基于 **“岗位职能”** 和 **“上下文隔离”** 两个核心原则来确立。以下是主流的划分方式和它们的边界线：

## 1. 核心子 Agent 的类型与边界划分

一个标准的 Coding Agent 团队通常由以下 4 类子 Agent 组成，它们各司其职，边界清晰：

### ① 路由与产品经理 Agent（Router / PM Agent）

- **职责边界**：负责“定义问题”。它直接对接用户输入的原始需求（比如：“帮我给这个 FastAPI 项目加一个 JWT 认证”）。它不做任何编码工作，只负责理解意图、拆解任务（Task Decomposition）、检查项目的现有技术栈。
    
- **输入/输出边界**：
    
    - **输入**：用户的模糊自然语言。
        
    - **输出**：拆解后的结构化任务列表（Todo List），并将其分发给下级 Agent。
        

### ② 架构与检索 Agent（Repo Navigator / RAG Agent）

- **职责边界**：负责 **“寻路与定位”**。大项目的代码量极大，不可能全部塞给模型。这个 Agent 的唯一任务是在代码库里“捞针”。它利用 AST（抽象语法树）、grep 工具或向量检索，找出修改这个需求**相关度最高的文件和函数。
    
- **输入/输出边界**：
    
    - **输入**：PM Agent 拆解的具体子任务。
        
    - **输出**：精准的代码上下文（例如：“修改 `auth.py` 的第 20-50 行，并参考 `config.py` 里的密钥配置”），**绝对不越界去改代码**。
        

### ③ 编码执行 Agent（Coder / Engineer Agent）

- **职责边界**：负责“局部编写”。它是纯粹的“码农”。它的上下文被严格限制在由检索 Agent 找出的那几个文件片断里。它只专注于逻辑实现，利用特定的 Write/Patch 工具对代码进行精准的增删改。
    
- **输入/输出边界**：
    
    - **输入**：精准的上下文 + 修改指令。
        
    - **输出**：改动后的代码补丁（Patch/Diff）。
        

### ④ 测试与调试 Agent（Tester / Debugger Agent）

- **职责边界**：负责“质量把关”**。它拥有运行环境（Sandbox/Docker）的控制权。它负责运行 `pytest`、`npm test` 或编译命令。如果报错，它会捕获报错日志，并拥有**“打回重写”的权力。
    
- **输入/输出边界**：
    
    - **输入**：Coder 刚改完的代码。
        
    - **输出**：测试通过报告（Pass）；或者精准的报错堆栈（Fail），并把错误推回给 Coder。
        

## 2. 子 Agent 之间的边界是如何在架构中“卡死”的？

在实际开发（如使用 LangGraph 编排）时，边界不能只靠 Prompt 维系，必须靠状态机（State Machine）和工具权限（Tool Governance）进行物理隔离：

### 维度一：工具权限的隔离（Tools Isolation）

这是划分边界最硬的手段。**Agent 能做什么，完全取决于你分给它什么工具。**

- **Coder Agent**：拥有 `view_file`、`edit_file` 工具，但**没有** `run_command`（执行命令）的权限。防止它一边写代码一边盲目运行。
    
- **Tester Agent**：拥有 `run_command`、`view_log` 工具，但**绝对没有** `edit_file` 的权限。它如果发现代码错了，自己不能动代码，必须通过状态机把错误信息转交给 Coder。
    

### 维度二：上下文的动态裁剪（Context Stripping）

如果把所有代码和聊天记录让所有 Agent 共享，边界就会模糊。

- 当状态从 RAG Agent 转移到 Coder Agent 时，系统会**抹去**大篇幅的代码搜索历史（Grep 日志），只把最终定位到的目标代码片段（Snippet）以结构化的形式传给 Coder。
    
- **这就好比**：PM 不需要看 Coder 写的每一行代码，Coder 也不需要知道 PM 和用户聊了多少个回合。
