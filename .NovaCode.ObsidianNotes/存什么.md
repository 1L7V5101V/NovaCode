因为代码里有一整套规则来约束"什么东西能进记忆、进去了怎么管"，而不是"所有东西都往里塞"。

具体看代码中控制"存什么"的机制：

| 控制手段 | 代码位置 | 效果 |
|---------|---------|------|
| **去重** — 同 text + 同 kind 只保留最新 | `memory.py:497-514` | 避免同一事实重复堆积 |
| **freshness 校验** — 文件改了，摘要自动失效 | `memory.py:275-279` | 保证记忆与文件系统一致 |
| **canonicalize** — 路径归一化 | `memory.py:265-272` | 同一文件不会存两份 |
| **显式 append_note** — 需要手动调 API 写入 | `memory.py:482-490` | 不会自动抓取一切 |
| **tag 分类** — 写入时打标签 | `memory.py:445` | 检索时可精确过滤 |
| **kind 区分** — process/episodic 等 | `memory.py:460-461` | 可针对不同类别做不同策略 |
| **显式失效** — `invalidate_file_summary` | `memory.py:488` | 代码显式管理生命周期 |

这些规则加起来，相当于给"存什么"圈了边界：**同一件事不会反复存、过期的会自动消失、路径不一致的会被归一、写什么由代码显式控制。**

相比之下，"什么时候取"没有一个类似的规则体系——它只能对当前 query 做 tag/关键词匹配，但你无法确保挑出来的 3 条就是对的。那是一个**预测问题**，不是一个**规则问题**。
