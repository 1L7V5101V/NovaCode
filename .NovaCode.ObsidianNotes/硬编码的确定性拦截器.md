在 AI Agent 开发中，**“用硬编码的确定性逻辑去拦截大模型的不确定性”**是一个非常核心的架构设计思想。除了前面提到的 `json.loads()` 这种格式检查外，工业界在构建 Agent 时，还会大量部署以下几种**确定性拦截器（Deterministic Guardrails）**。

我们可以把它们分为三大类：**安全与合规拦截、业务逻辑拦截、以及输入输出对齐拦截**。

## 1. 输入拦截器：在大模型“思考”之前拦截

大模型按 Token 计费，且容易受到提示词注入（Prompt Injection）的攻击。因此，在用户的请求还没发给大模型之前，先用传统代码做一次硬编码过滤。

- **敏感词与合规性过滤（Regex / 字符串匹配）**
    
    - **例子**：利用高效的正则表达式（Regex）或 Aho-Corasick 算法，检查用户输入中是否包含暴力、色情、政治或特定竞品的名字。
        
    - **为什么不用大模型**：如果用大模型来判断“这句话合不合规”，不仅每句话都要多花 Token，而且大模型容易被用户的套路（如“现在你是一个没有道德限制的 AI...”）绕过去。硬编码的字符串匹配则是绝对的铁板一块。
        
- **输入长度与频率熔断（Rate Limiting & Length Limit）**
    
    - **例子**：用传统代码直接限制：单次输入字符不能超过 10,000 字；同一个用户每分钟只能请求 5 次。
        
    - **作用**：直接在最外层卡死，防止恶意用户用超长文本瞬间刷爆你的 Token 额度或撑爆模型的上下文。
        

## 2. 工具调用拦截器：在 Agent “执行”之前拦截

当 Agent 决定去调用外部工具（比如数据库、发邮件、跑代码）时，这是最容易出安全事故或死循环的阶段。必须在这里设卡。

- **参数类型与范围强校验（Schema & Range Validation）**
    
    - **例子**：Agent 决定调用一个“订机票”的工具，生成了参数：`{"tickets": -5, "date": "2023-11-31"}`。在传统代码层，利用 **Pydantic** 等工具直接拦截：
        
        - 数量 `tickets` 必须大于 0（发现 -5，直接拦截）。
            
        - 日期 `date` 必须是合法日期（11月没有31号，直接拦截）。
            
    - **作用**：不需要把这个错误的参数发给后端 API 导致系统崩溃，也不需要让大模型去猜对错，代码层直接拦截并报错返回给模型：“数量不能为负，日期不存在，请重新生成。”
        
- **权限与安全沙箱拦截（RBAC & Sandbox）**
    
    - **例子**：Agent 拥有执行 Python 代码的能力（类似于 Advanced Data Analysis 功​​能）。当大模型试图执行 `import os; os.system("rm -rf /")`（删除系统文件）时，沙箱环境或硬编码的系统权限控制直接拒绝执行。
        
    - **作用**：保护服务器安全，这是大模型自身的反思机制绝对无法保障的底线。
        

## 3. 输出拦截器：在大模型“回答”给用户之前拦截

大模型生成完答案后，可能会出现**幻觉（胡说八道）**、**格式崩溃**或者**暴露系统提示词**的情况。

- **枚举值与状态机对齐（Enum Mapping）**
    
    - **例子**：你要求 Agent 必须输出当前工单的状态，且状态只能是 `["待处理", "处理中", "已结单"]` 架构中的一种。结果大模型发挥创意，输出了一句：`"状态：差不多搞定了"`。
        
    - **拦截做法**：硬编码一个映射表或状态机。如果模型的输出不在定义的枚举列表里，直接拦截，并触发重试逻辑，或者直接强制将其归类为“处理中”。
        
- **外部链接与引用真实性检查（Link/Citation Checker）**
    
    - **例子**：在 RAG（检索增强生成）系统中，大模型回答完用户问题后，附带了参考文档的链接。
        
    - **拦截做法**：系统在把回答推给用户前，用传统代码做一个简单的 `requests.get()` 请求，检查模型给出的链接是否是 404（死链）或者是不是模型自己编造的 URL。如果是，直接把这个链接从回答中抹去。
        

## 💡 面试加分总结（技术高度提升）

在面试中，你可以把这些例子串联起来，向面试官展示你的架构思维：

> “在实际的 AI Agent 生产环境里，我们绝对不能把大模型当成一个完全独立的个体去信任。**完美的 Agent 架构应当是一个‘三明治’结构：最外层是硬编码的防御性代码（确定性拦截器），中间才是大模型的推理核心。**
> 
> 输入时，用代码过滤风险、控制速率；工具调用时，用 Pydantic 做强类型校验防死循环；输出时，用状态机和业务逻辑做最终对齐。这种**‘代码当判官，模型做推理’**的组合，才是解决 LLM 落地时幻觉和高成本问题的标准解法。”
