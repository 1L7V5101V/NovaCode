Pico 用了多层机制来阻断"幻觉循环"的自增强效应：

### 第 1 层：工具执行前 → 阻断重复行为

```python
def repeated_tool_call(self, name, args):
    tool_events = [item for item in self.session["history"] if item["role"] == "tool"]
    if len(tool_events) < 2:
        return False
    recent = tool_events[-2:]
    return all(item["name"] == name and item["args"] == args for item in recent)
```

如果 agent 连续两次调了**同名同参数**的工具，第三次直接被拒，返回错误。

### 第 2 层：工具执行中 → 结果 vs 假设严格分离

风险工具执行前后都会捕获工作区快照（`runtime.py:1078-1083`）：

```python
before_snapshot = self.capture_workspace_snapshot()
result = clip(tool["run"](args))
after_snapshot = self.capture_workspace_snapshot()
affected_paths, diff_summary = self.diff_workspace_snapshots(before_snapshot, after_snapshot)
```

agent 的"错误代码"被写进文件，但工具执行结果（diff、affected_paths、exit_code）是**从真实文件系统读取**的，而不是从模型中提取的。错误代码本身不会成为"模型认为文件状态是什么"的证据——真实状态来自文件系统的 SHA-256，而不是来自模型的输出。

### 第 3 层：工具执行后 → 错误作为"提醒"而非"事实"

```python
if status == "error":
    text = f"{name} error on {path_text}; check the failure before retry"
```

错误被记成 process note，但内容是 **"检查失败原因后再重试"**，而不是"config.txt 的内容是 XXX"。后者的错误版本（"config.txt 的内容是 YYY"）不会被存储为知识。

### 第 4 层：全局退出 → 不会无限循环

```python
max_attempts = max(self.max_steps * 3, self.max_steps + 4)
```

超过重试上限直接停止，不会让模型在错误上无限叠加。

### 第 5 层：恢复时的 freshness 校验 → 跨会话打断循环

如果会话恢复时发现文件摘要的 SHA-256 和当前文件不一致，摘要自动失效。这意味着即使上一轮产生了错误代码，下一轮恢复时记忆不会保留"错误文件内容摘要"。

---

**一句话：Pico 不信任模型对状态的描述，只信任工具执行前后从文件系统捕获的快照差异。幻觉循环会在"重复调用被拒"或"工具执行结果覆盖假设"这两个环节被打断。**
