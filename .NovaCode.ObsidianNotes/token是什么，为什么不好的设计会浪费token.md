
## 1. 什么是 Token？

简单来说，**Token 是大语言模型（LLM）处理文本的最小语义单位**。

- **文本的转化**：模型在接收输入（Prompt）或生成输出（Completion）时，并不会直接读取原始的字符串，而是通过 Tokenizer（分词器）将文本切分成一个个片段。
    
- **表现形式**：一个 Token 可以是一个完整的单词（例如英文中的 `"apple"`）、一个单词的一部分（如 `"pre"` 或 `"ing"`），或者一个汉字/标点符号。通常情况下，100 个英文单词大约对应 130-140 个 Token，而由于分词机制的不同，一个汉字可能会占用 1 到多个 Token。
    

在开发 Agent 时，Token 不仅是模型理解世界的“像素”，更是评估**上下文窗口（Context Window）容量**和**API 计费**的直接指标。

## 2. 为什么不好的设计会浪费 Token？

AI Agent 与传统单次对话（Chat）不同，它往往涉及**长周期记忆、多轮工具调用（Tool Calling）和复杂的反思循环（Reflection Loop）**。如果系统架构设计得不够精细，很容易导致 Token 的指数级浪费，主要体现在以下几个典型场景：

### ① 缺乏上下文管理（Context Inflation）

- **问题设计**：有些不成熟的 Agent 设计会把所有历史对话、长文本知识库内容、完整的 API 接口文档，一股脑地全部塞进每一轮对话的 Prompt 中。
    
- **后果**：随着对话轮数的增加，Prompt 的体积呈线性甚至指数级膨胀。LLM 在处理长上下文时，每一轮对话都要重新计算整段输入。如果 Agent 只需要回答一个“是”或“否”，却被迫读了 20k tokens 的冗余背景，这就是极大的浪费。
    

### ② 工具调用（Tool Calling）的死循环与低效重试

- **问题设计**：在 RAG（检索增强生成）或 Tool-use 场景下，如果 Agent 的[[反思边界]]（Critique Loop）或错误处理机制写得太宽泛，当模型生成的参数不符合工具要求时，Agent 可能会陷入 *“出错 -> 报错给模型 -> 模型再次盲目尝试 -> 再次报错”* 的死循环。
    
- **后果**：每一次错误的重试都在消耗大量的输入和输出 Token。如果缺乏最大步数限制（Max Loops）或硬编码的强验证，一次用户请求可能会在后台悄悄烧掉几十万 Token。
    

### ③ 原始数据未经过清洗与结构化（Raw Data Dumping）

- **问题设计**：从向量数据库（Vector DB）检索出 Top-K 的文档片段后，或者利用爬虫抓取网页后，不经过任何预处理（如清洗 HTML 标签、去重、核心段落提取），直接作为上下文投喂给模型。
    
- **后果**：数据中夹杂大量的广告、空白字符、无意义的符号，这些非关键信息会无形中吞掉大量的 Token 份额，还会稀释模型的注意力，导致 Needle in a Haystack（大海捞针）能力下降。
    

### ④ 过于臃肿的 System Prompt 与 Schema 定义

- **问题设计**：为了让 Agent 遵循特定格式，给工具（Tools）定义了极其复杂的 JSON Schema，或者在 System Prompt 中写了大量重复、冗长的行为准则。
    
- **后果**：System Prompt 是每轮对话都要固定的“底噪”。如果基底 Prompt 过于臃肿，在多轮交互中被反复复制传递，哪怕用户只说了几个字，系统也在承受巨大的隐性 Token 开销。
    

## 3. 在我过往的 Agent 项目中，是如何优化这一点的？

针对这些痛点，我在开发 Agent 应用时，通常会采取以下针对性的优化策略：

- **引入持久化与裁剪的记忆机制**：使用 LangGraph 等框架进行精细的状态管理，不使用全局透传，而是采用 **Summary Memory（摘要记忆）** 或 **Buffer Window Memory**，动态滑动裁剪历史记录。
    
- **精简的工具与 Prompt 路由（Routing）**：不把所有工具一次性塞给模型，而是通过意图识别，先路由到特定的子任务节点（Sub-agent），每个节点只暴露其所需的最小工具集和最精简的 Prompt。
    
- **结构化输出与 Pydantic 强校验**：在工具调用前置拦截非法输入，利用代码层面的强验证代替模型的“肉眼校对”，避免将琐碎的报错信息交由模型去理解和重试，从而切断死循环。
    
- **利用大模型新特性**：在支持 Context Caching（上下文缓存）的模型上，合理设计 Prompt 结构，将不常变动的系统设定、长知识库沉淀在缓存中，将首字延迟和计算费用降低数倍。
    

**总结：** 在 Agent 开发中，**“Token = 算力 = 钱 + 延迟”**。优秀的 Agent 设计应当像优秀的程序员管理内存一样去精益求精地管理 Token，用最小的上下文，撬动模型最核心的推理能力。
