这里的 `transcript` 指的是 **完整对话/执行历史记录**，也就是 agent 一路做任务时留下来的流水账：

- 用户说了什么
- assistant 回答了什么
- 调用了哪些工具
- 工具返回了什么结果
- 读了哪些文件
- shell 命令输出了什么
- 中间推理过程留下了哪些历史片段

所以“最容易膨胀的一层，其实是 transcript”意思是：

> 在多轮代码任务里，最容易把上下文撑爆的，不一定是当前用户请求，也不一定是 memory，而是历史记录本身。因为每一轮对话、每一次工具调用、每一次文件读取、每一段命令输出，都会不断追加到 transcript 里。

比如 agent 连续做几轮：

1. 读了一个 300 行文件；
2. 跑了一次测试，输出一大段日志；
3. 又读了两个文件；
4. 修改失败后又跑了一次命令；
5. 用户补充需求；
6. assistant 又解释了一段。

这些东西如果都原样保留，`transcript` 很快就会变得特别长。

所以文档后面才说，Pico 对 history/transcript 做了新旧分层：

> 最近 6 条保留更多细节，旧历史压成很短摘要，重复 `read_file` 按路径去重。

面试里可以这样解释：

> transcript 就是 agent 的历史流水账。它最容易膨胀，因为代码任务里每轮都会追加工具输出、文件内容、命令日志和对话记录。如果不处理，prompt 很快会被旧历史占满。所以 Pico 对 transcript 做了新旧分层压缩，最近的内容保留更多细节，旧内容只保留摘要和关键事实。
