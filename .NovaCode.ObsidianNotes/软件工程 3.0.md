## 软件工程三个时代的演进

|                     | 1.0 传统软件工程 | 2.0 敏捷软件工程    | 3.0 智能软件工程       |
| ------------------- | ---------- | ------------- | ---------------- |
| 编程范式                | 结构化 → 面向对象 | 面向对象 + DevOps | 面向大模型            |
| 开发流程                | 瀑布、重文档     | 迭代、轻文档        | 人机协作、自然语言驱动      |
| 核心资产                | 代码         | 代码 + 流程       | 模型 + 数据 + Prompt |
| 质量保障                | 人工测试       | 自动化测试         | 智能化测试            |

## 软件工程 3.0 的核心范式转变

### 1. SaaM：Software as a Model

软件工程 3.0 的核心特征是"软件即模型"（SaaM）——软件开发过程被看作人与计算机之间自然交互的过程，开发者利用大模型进行需求分析、设计、编程和测试，强调数据、模型以及提出有效问题的价值。

这是个很根本的认知转变：**代码不再是交付物，而是大模型的输出产物**。真正的资产变成了：

- 你的需求描述能力（Prompt Engineering）
- 你的领域知识（数据 + 约束）
- 你的验证能力（知道模型输出对不对）

### 2. 模型驱动研发

大模型不再仅仅是辅助工具，而是成为软件开发的核心驱动力，参与到需求分析、设计、代码生成、测试等各个环节；人负责更高层次的架构设计、问题定义和创新，AI 模型负责执行重复性、模板化的任务。

### 3. 核心能力重构

3.0 时代的核心能力建设包括：提示工程、RAG、智能体、数据治理、模型工程和安全治理，覆盖需求分析、架构设计、UI 生成、结对编程、测试智能化和运维监控等关键环节。

### 4. 生产关系的变化

"超级个体与新型团队"——个人借助 AI 能撬动过去需要整个团队才能完成的工作，团队协作模式也因此重组。

## 我对这个框架的判断

**真实发生的部分：**

"软件工程 3.0"描述的很多变化已经在实际发生，但我觉得有几个洞察更精准：

- **抽象层级在上移**——程序员的工作从"写循环"到"写函数"到"写模块"到现在"描述意图"，这是真实趋势
- **验证能力变得比生成能力更值钱**——谁能判断 AI 输出的对错，谁就掌握了新的核心竞争力
- **调试范式改变**——以前调代码，现在调 Prompt + 数据 + 模型行为，工具链还远没有跟上

**我会质疑的部分：**

也有人指出挑战所在：我们管理的将不再是可读的代码，而是行为难以预测的"黑箱"模型，多人、多模型之间的协同、调试和版本控制将成为全新的难题；当企业核心代码和业务数据被用于模型训练时，数据隐私和知识产权的保障也是待解问题。

这些挑战被书里提到了，但没有被充分解决——**可解释性、可审计性、责任归属**在 3.0 范式里反而比 2.0 更难处理。

## 对你的研究的关联

有趣的是，你做的 FEM 仿真工作天然契合 3.0 范式里的一个核心矛盾：

**物理仿真领域的"黑箱"问题比纯软件工程更严重**——AI 生成的代码逻辑错了最多 bug，但 AI 辅助的 FEM 参数或边界条件错了可能给出"看起来合理"的错误结果，而你的物理直觉是唯一的验证屏障。

所以在你的场景里，3.0 范式里"人负责验证"这件事的权重比普通软件开发高得多。
