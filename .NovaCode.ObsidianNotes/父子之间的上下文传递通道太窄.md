## 问题在哪

**说"没有独立窗口"不够准确**——子 agent 实际上是全新的 `Pico` 实例，有自己的 session、自己的 context manager，**这本身就是独立窗口**。

真正的薄弱点是**父子之间的上下文传输**：

| | Pico | Claude Code |
|---|---|---|
| **父→子** | 300 字符的裁剪字符串，无结构 | 完整 task 描述 + 可选的父上下文摘要 |
| **子→父** | 子返回原始文本，直接拼入父 history | 子返回 **1000-2000 token 的结构化摘要**，父 context 几乎不膨胀 |
| **增量更新** | 无 | **delta 摘要**（在已有摘要基础上只追加新消息的 1-2 句） |
| **独立工作能力** | 子可自由读文件、搜索，不污染父 context（✅） | 同样独立窗口（✅） |
| **关键文件/状态恢复** | 无 | 子完成后可重新注入最近文件 |

最核心的差距是：pico 的子 agent **从父 agent 那里几乎得不到有用上下文**——300 字符只够塞一两句历史摘要，不够传递当前任务状态、已读过的文件、已做的决策。子 agent 基本是从零开始重新探索。而 Claude Code 的子 agent 虽然也独立，但父 agent 可以通过 task 描述准确传递意图，且子 agent 返回的结果是轻量摘要，不会让父 context 膨胀。

更关键的是 **pico 没有 delta 增量摘要机制**——Claude Code 对长时间运行的背景子 agent 使用增量摘要，每次只合并新消息；pico 没有这种能力。

## 纠正

我之前的说法有偏差，应该修正为：**子代理有独立窗口，但父子之间的上下文传递通道太窄（300 字符无结构截断），且缺乏增量摘要和结构化 handoff 机制。**
