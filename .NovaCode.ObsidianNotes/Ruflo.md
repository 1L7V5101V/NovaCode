---

## Ruflo 是什么

Ruflo 前身是 Claude Flow，是一个专门为 Claude Code（以及 OpenAI Codex）设计的 AI 编排框架，核心目标是把 agentic coding assistant 扩展成具备 ~100 个专业化 agent 并行协作的 multi-agent 平台，通过智能路由、共享内存和自学习工作流来处理复杂软件任务。

底层由 Rust 驱动的 AI 引擎（Cognitum）支撑，包含 embeddings、内存系统和插件系统。

---

## 核心特性

主要能力包括：基于 swarm 的分层协调与共识机制；从历史执行中自学习、动态路由到最优 agent 的自适应机制；结合向量检索（HNSW）、共享内存和知识图谱的跨会话持久记忆；以及用 WASM + LLM 多层路由降低延迟、声称可降低约 75% API 成本的优化机制。

安全层面，内置 Rust 编写的 guidance-kernel 做实时拦截：PII 过滤防止 agent 泄露个人信息、Prompt Injection 检测、以及生成依赖的 CVE 自动扫描。

---

## 实际应用场景

Ruflo 的并行 agent 架构使得在单次协调会话中重构 20 万行代码成为可能——queen agent 将重构任务分解为文件级工作流，分配给各 worker agent，再将结果合并成一个保持全局一致性的 diff。

---

## 我的判断

Ruflo 的设计方向和我们刚才讨论的思路有很多共鸣——分层编排、共享状态、持久记忆都对上了。但有几点我会保持审慎：

**值得关注的地方：**

- Rust 底层引擎 + WASM 路由，性能优先的架构选择是认真的
- 跨机器、跨信任边界的联邦通信，这是真正面向企业场景的

**我会质疑的地方：**

- "~100 个并行 agent"这个数字更像是营销说法——按我们刚讨论的 Amdahl's Law，真实任务里能有效利用的并行度远没有这么高
- "降低 75% API 成本"的说法需要看具体 benchmark，WASM 路由本质上是用本地规则替代 LLM 调用，适用场景有限
- 项目很新（2026 年初才改名），27k stars 增长很快，但生产稳定性还需要时间验证

你是在考虑把它用到某个具体项目里，还是纯粹在研究它的架构设计？
