## 边际收益递减：为什么？

### 根本原因：任务本身有串行下界

任何任务都可以分成**可并行部分**和**必须串行部分**。

这就是 **Amdahl's Law** 的核心：

$$T(n) = T_{serial} + \frac{T_{parallel}}{n}$$

加速比上界：

$$S(n) = \frac{1}{f + \frac{1-f}{n}} \xrightarrow{n \to \infty} \frac{1}{f}$$

其中 $f$ 是串行部分占比。

**即使串行部分只有 20%，无论加多少 agent，加速比永远不超过 5×。**

在代码任务里，串行下界来自：

- 架构决策必须先做，实现才能开始
- B 模块依赖 A 模块的接口定义
- 测试必须等实现完成
- 最终集成只能有一个 agent 做

所以并行 agent 增加到一定数量后，大家都在**等关键路径上的那个串行节点**，多出来的 agent 是空转的。

## 协调成本递增：为什么？

这里有两个叠加效应：

### 效应一：通信链路是 O(n²)

n 个 agent 两两之间潜在的通信需求是：

$$\text{链路数} = \frac{n(n-1)}{2}$$

|Agent 数|潜在链路|
|---|---|
|2|1|
|5|10|
|10|45|
|20|190|

即使用分层编排压缩成 O(n)，**每条链路本身的信息损耗也在累积**——agent 之间传递的不是原始代码，而是经过压缩的"描述"，每经过一跳就有语义损失。

### 效应二：共享状态的竞争是超线性的

多个 agent 并发读写共享状态时，冲突概率不是线性增长的。

设每个 agent 在单位时间内写操作概率为 $p$，n 个 agent 中**至少发生一次冲突**的概率：

$$P_{conflict} = 1 - (1-p^2)^{\binom{n}{2}} \approx 1 - e^{-\binom{n}{2}p^2}$$

随 n 增大，冲突概率趋近于 1——而每次冲突都需要协调，协调本身消耗时间和 context。

### 效应三：Orchestrator 成为新瓶颈

Orchestrator 需要追踪所有 worker 的状态，其认知负担是 O(n) 的——但它自己也是一个有 context 限制的 LLM。

当 n 增大时，Orchestrator 的 context 被状态追踪占满，用于"真正思考"的空间反而变少，**决策质量下降**，这又反过来增加 worker 的出错率。

## 两条曲线放在一起

```
收益/成本
    ↑
    │         ← 边际收益（递减）
    │    ╱‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾
    │   ╱
    │  ╱              ╱← 协调成本（递增）
    │ ╱           ╱
    │╱        ╱
    ├────────────────────→ Agent 数量
    0    n*
         ↑
      最优并行度
```

__n_ 就是净收益最大的点_*——超过这个点，加 agent 是负收益的。

所以实际系统设计里，正确的问题不是"能并行多少"，而是**"这个任务的串行下界是多少，值得并行的部分有多大"**——先算清楚这个，再决定开几个 agent。
